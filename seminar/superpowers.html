프레이밍

왜 방법론 레이어인가

런타임이 "어떻게 API를 호출하는가"라면, 방법론 레이어는 "에이전트가 어떤 절차로 생각해야 하는가"다.

  • Claude Code, Codex, Gemini CLI — 런타임은 달라도 방법론은 동일하게 동작
  • 스킬 파일(.md)이 런타임과 독립 — 11개 하니스에서 그대로 읽힌다
  • 사용자 관점: "설치만 하면 알아서 발동"
  • 빌더 관점: 네 개의 명확한 레이어가 협력한다
    • 부트스트랩 훅 → 라우터 스킬 → 스킬 라이브러리 → 방법론 파이프라인
  • 핵심 통찰: Superpowers는 "툴 설치"가 아니라 에이전트의 의사결정 구조를 덮어쓰는 방법론 주입

Module 2 · 스킬 해부학

description이 트리거 신호다

에이전트는 스킬 본문을 모두 읽지 않는다. description만 먼저 스캔해 로드 여부를 결정한다.

  • name — 문자·숫자·하이픈만. 괄호·특수문자 금지
  • description — "Use when…"으로 시작. 워크플로 요약 금지
    • 요약이 들어가면 에이전트가 본문을 건너뛰는 "shortcut"이 생긴다
  • 스킬 3종: Technique(단계) · Pattern(사고방식) · Reference(API 문서)
  • flat namespace — skills/ 아래 중첩 없이 배치
  • 지원 파일은 100줄+ Heavy reference 또는 실행 가능 도구일 때만 분리

Module 4 · 방법론 파이프라인

세 관문이 품질을 결정한다

게이트의 수가 품질의 수준을 결정한다. 게이트가 없으면 에이전트는 가정을 검증하지 않은 채 수백 줄을 만들어낸다.

  • 설계 게이트 — brainstorming HARD-GATE: 승인 전 코드 작성 절대 금지
    • "Simple" projects are where unexamined assumptions cause the most wasted work.
  • 플랜 게이트 — No Placeholders: "TBD", "적절한 에러 처리 추가" 전부 plan failures
  • 실행 게이트 — 블로커에서 즉시 멈추기. 추측으로 앞으로 가는 것은 진전이 아니다

05
모든 것에 TDD

Red-Green-Refactor는 코드에만 적용되지 않는다 — 스킬 문서 작성에도 같은 사이클을 강제하는 메타 TDD

Module 5 · TDD

TDD의 두 층위

코드 TDD

Iron Law

NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST

코드를 먼저 작성했으면? 삭제하고 처음부터. "참조용 유지"도, "적응"도 없다.

스킬 TDD (메타)

Writing Skills IS TDD

서브에이전트 압박 시나리오 = 테스트 케이스

SKILL.md = 프로덕션 코드

에이전트가 스킬 없이 실패하는 것을 보지 않았다면, 스킬이 옳은 것을 가르치는지 알 수 없다.

Module 6 · 서브에이전트

컨텍스트 분리가 전부다

서브에이전트는 오케스트레이터의 세션 히스토리를 절대 상속받지 않는다.

  • 전달하는 것: brief 파일 경로 · 이전 태스크 인터페이스 결정 · 글로벌 제약 · 보고서 경로
  • 전달하지 않는 것: 이전 태스크 구현 요약 · 세션 전체 히스토리
  • 병렬화 조건: 각 문제가 다른 루트 원인 · 같은 파일 편집 없음 · 순서 의존 없음
  • 한 응답에 여러 dispatch = 병렬 실행. 응답당 하나 = 순차
  • SDD는 구현 서브에이전트 병렬 dispatch 금지 — 별도 스킬(dispatching-parallel-agents)의 영역
07
리뷰 & 검증 루프

증거 없는 완료 주장 금지 — 구현 이후의 게이트 루프가 신뢰를 만든다

Module 7 · 리뷰 & 검증

증거를 요구하는 구조

에이전트의 "완료"를 신뢰할 수 있게 만드는 것은 성실함이 아니라 증거를 요구하는 구조다.

  • requesting-code-review — "Review early, review often": 태스크 완료 후·주요 기능 후·main 머지 전
  • receiving-code-review — 기술적 평가, 감정적 수행 아님. "You're absolutely right!" 금지
  • verification-before-completion — Iron Law: NO COMPLETION CLAIMS WITHOUT FRESH VERIFICATION EVIDENCE
  • finishing-a-development-branch — 테스트 통과 → 환경 감지 → 옵션 제시 → 실행 → 정리

08
행동 형성 언어 기법

스킬은 산문이 아니라 행동을 형성하는 코드다 — 같은 정보도 어떻게 쓰느냐가 에이전트 준수율을 바꾼다

Module 8 · 행동 형성

다섯 가지 설계 기법

94% PR 거부율 환경에서 에이전트를 신뢰할 수 있게 만드는 방법.

  • 강조 프레이밍 — EXTREMELY-IMPORTANT · HARD-GATE: 대문자 XML 태그는 훈련 데이터에서 높은 우선순위 신호
  • 합리화 차단 — Red Flags 표: 12가지 회피 사고를 미리 이름 붙이고 "STOP" 판정을 미리 짝지음
  • 용어 선택 — "your human partner": "user"가 아닌 파트너 → 책임감 기준선이 달라진다
  • 구조적 강제 — dot-graph 플로우차트: 건너뜀이 구조적으로 불가능한 의사결정 경로
  • eval 기반 튜닝 — adversarial pressure testing: 스킬 파일 = 소스 코드, eval = 테스트 스위트
