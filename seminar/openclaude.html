전체 아키텍처 조감도

6개 레이어, 그리고 레이어 사이의 "모름"이 만드는 격벽

openclaude 레이어 구조

flowchart TD
    A["CLI 진입점\nsrc/entrypoints/cli.tsx"]
    B["Bootstrap / STATE 싱글턴\nsrc/bootstrap/state.ts"]
    C["main.tsx — 초기화 오케스트레이터"]
    D["QueryEngine — 에이전트 루프\nsrc/QueryEngine.ts"]

    subgraph TOOLS["툴 레지스트리  src/tools/"]

    subgraph SVC["서비스 레이어  src/services/"]

    UI["Ink UI  src/components/"]

    A -->|"동적 import"| B
    A -->|"await main()"| C

Module 0 · 핵심 통찰

분리의 원칙

레이어가 "모르는" 것

의존성 역전

QueryEngine은 툴이 무엇인지 모른다.
QueryEngineConfig로 주입받은 배열만 호출할 뿐이다.

이 덕분에 SDK 경로와 REPL 경로가 같은 엔진을 공유한다.

Bootstrap 설계

import DAG의 leaf

state.ts는 다른 src 파일을 절대 임포트하지 않는다.

모든 레이어가 순환 없이 STATE를 읽을 수 있는 이유.

01
한 턴의 전체 흐름

엔터를 누른 순간부터 최종 텍스트가 출력될 때까지

한 턴의 시퀀스

sequenceDiagram
    participant U as User
    participant QE as QueryEngine
    participant QL as queryLoop()
    participant LLM as Provider API
    participant T as Tool

    U->>QE: submitMessage(prompt)
    QE->>QL: query({ messages, systemPrompt })

    loop 스텝 반복
        QL->>LLM: callModel()
        LLM-->>QL: stream: text delta
        LLM-->>QL: stream: tool_use 블록
        QL->>QL: needsFollowUp = true
        QL->>T: tool.call(input)
        T-->>QL: ToolResult
        QL->>QL: messages = [...msgs, assistantMsg, toolResult]
    end

    QL-->>QE: reason: completed
    QE-->>U: yield { type:'result', result: textResult }

한 턴 = 1회 submitMessage(). 스텝 = while(true) 이터레이션 1회. 툴 콜마다 스텝이 추가된다.

02
에이전트 루프

자율 반복의 심장부 — 상태머신, 종료 조건, auto-continuation

루프 상태머신

flowchart TD
    ENTRY["루프 진입\nwhile(true)"]
    STREAM["LLM API 스트리밍"]
    NEEDS_TOOL{"needsFollowUp?"}
    EXEC_TOOL["runTools() 실행"]
    MAXTURN{"maxTurns 초과?"}
    NEXT_ITER["다음 이터레이션\nstate = next, continue"]
    AUTO_CONT["auto-continuation 검사\nanalyzeContinuationIntent()"]
    NUDGE{"nudge 필요?"}
    DONE["루프 종료\nreason: completed"]
    ABORT["루프 종료\nreason: aborted_*"]
    MAX_END["루프 종료\nreason: max_turns"]

    ENTRY --> STREAM
    STREAM --> NEEDS_TOOL
    NEEDS_TOOL -->|"yes"| EXEC_TOOL
    NEEDS_TOOL -->|"no"| AUTO_CONT
    EXEC_TOOL --> MAXTURN
    MAXTURN -->|"yes"| MAX_END
    MAXTURN -->|"no"| NEXT_ITER
    NEXT_ITER --> ENTRY
    AUTO_CONT --> NUDGE
    NUDGE -->|"yes"| NEXT_ITER
    NUDGE -->|"no"| DONE
    STREAM -->|"abort 신호"| ABORT

    style DONE fill:#1a4731,color:#fff
    style ABORT fill:#4a1a1a,color:#fff
    style MAX_END fill:#4a2a1a,color:#fff

auto-continuation: LLM이 툴 없이 "이제 X를 하겠다"고만 쓰면 nudge 메시지를 주입하고 재진입. 상한: MAX_CONTINUATION_NUDGES = 20

Module 3 · 툴 생명주기

스키마 → 권한 → 실행 → 직렬화

Tool 인터페이스 계약

buildTool()

  • inputSchema — Zod 스키마 = LLM의 API 문서

  • mapToolResultToToolResultBlockParam() — 직렬화

컨텍스트 관리 — 세 겹의 방어선

flowchart LR
    A["매 턴\n시작"] --> B["토큰 추정\nbytes / 4"]
    B --> C{"임계 판정"}

    C -->|"95% 이상\nCollapse 활성"| D["Context Collapse\n스팬만 접기\n→ placeholder 교체"]
    C -->|"Auto-compact\n임계 초과"| E["세션 메모리\n먼저 시도"]
    C -->|"정상"| K["API 호출"]

    E -->|"있음"| F["SessionMemory\n파일로 교체"]
    E -->|"없음"| G["forked agent\n요약 생성"]
    G --> H["summary +\nattachments 재구성"]

    D --> K
    F --> K
    H --> K

200k 윈도우 기준: 200k − 20k(요약 예약) − 13k(버퍼) = 167k 토큰에서 auto-compact 트리거. p99.99 측정값으로 산출.

06
서브에이전트 오케스트레이션

부모 루프 안에서 자식 루프가 돌아간다

서브에이전트 스폰 흐름

sequenceDiagram
    participant P as 부모 에이전트 루프
    participant AT as AgentTool.call()
    participant W1 as Worker 1
    participant W2 as Worker 2

    P->>AT: Agent({ run_in_background:true, prompt:"..." })
    AT->>AT: registerAsyncAgent()
    AT-->>P: { status:"async_launched", agentId:"a1b" }

    P->>AT: Agent({ run_in_background:true, prompt:"..." })
    AT-->>P: { status:"async_launched", agentId:"x7q" }

    par Worker 1
        AT->>W1: query() — 새 QueryEngine 루프
        W1-->>AT: 완료
        AT->>P: task-notification(a1b, completed)
    and Worker 2
        AT->>W2: query() — 새 QueryEngine 루프
        W2-->>AT: 완료
        AT->>P: task-notification(x7q, completed)
    end

    P->>P: 결과 합성

서브에이전트 = 또 하나의 QueryEngine 루프. 부모와 분리된 AbortController → ESC를 눌러도 백그라운드 에이전트는 계속 실행.

확장성: MCP · Skills · 슬래시 커맨드

코어를 건드리지 않고 기능을 확장하는 세 가지 축

Module 7 · 확장 세 축

런타임 등록

  • MCP 서버 — JSON-RPC로 외부 프로세스가 툴을 노출
    • 이름 형식: mcp__{server}__{tool}
    • 기본값: passthrough → 매번 사용자 승인 요청
  • Skills — Markdown 파일이 슬래시 커맨드가 된다
    • .claude/skills/*.md → YAML 프런트매터 파싱
    • 모델이 SkillTool로 호출 또는 사용자가 /skill로 직접 호출
  • MCP Skills 보안 가드
    • hooks: undefined 강제 — 원격 쉘 실행 차단
    • allowedTools: [] 강제 — 자동 승인 우회 차단
08
권한 & 안전장치

자율 에이전트가 파일·셸에 접근할 때 신뢰를 유지하는 방법

7단계 권한 판정 파이프라인

flowchart LR
    A["툴 호출\n요청"] --> B["1a. deny 규칙?"]
    B -->|"매칭"| DENY["차단"]
    B -->|"없음"| C["1c. tool.checkPermissions()"]
    C -->|"deny"| DENY
    C -->|"ask + safetyCheck\n.git/.ssh/.claude"| ASK["사용자 승인\n요청"]
    C --> D["2a. 모드 확인"]
    D -->|"bypassPermissions\n또는 fullAccess"| ALLOW["통과"]
    D -->|"그 외"| E["2b. alwaysAllow 규칙?"]
    E -->|"매칭"| ALLOW
    E -->|"없음"| F["passthrough → ask\n다이얼로그"]
    F -->|"Yes"| ALLOW
    F -->|"No"| DENY

    style DENY fill:#5f1e1e,stroke:#ff4a4a,color:#ffe0e0
    style ALLOW fill:#1e5f3a,stroke:#4aff9a,color:#e0ffe8
    style ASK fill:#1e3a5f,stroke:#4a9eff,color:#e0f0ff

모르면 막는다 — passthrough는 ask로 격상. deny 규칙과 safetyCheck는 bypassPermissions에서도 우회 불가.
