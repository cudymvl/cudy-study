Module 2

에이전트 루프

자율 반복의 심장부 — 스트리밍 파싱, 툴콜 누적, 종료 판정, auto-continuation의 내부 작동

왜 이 개념인가

Module 1에서 "한 턴"의 전체 흐름을 조감도 수준으로 살펴봤다. LLM이 툴 콜을 돌려주면 루프가 재진입한다는 사실도 확인했다. 이번 모듈은 그 루프 자체의 내부 구조를 해부한다.

"에이전트"라는 단어는 종종 막연하게 사용되지만, 구현 관점에서는 명확하다. 에이전트 = 자율 루프(autonomous loop). 외부에서 다음 입력이 들어오길 기다리는 챗봇과 달리, 에이전트는 자신의 출력(툴 결과)을 스스로 다음 입력으로 만들어 반복 실행한다. 루프가 없으면 에이전트가 아니라 단발 호출(one-shot call)이다.

구분단발 호출에이전트 루프
입력 공급자 사람 (매번) 루프 자체 (툴 결과 → 다음 입력)
LLM 호출 횟수 / 요청 1회 1~N회 (툴 콜 수 + 1)
종료 시점 결정권 없음 (즉시 종료) 루프가 종료 조건을 판단
대표 코드 경로 단순 fetch queryLoop() + while (true)

openclaude의 루프는 src/query.ts의 queryLoop() 함수다. while (true)를 중심으로 상태 머신처럼 동작하며, 각 이터레이션은 "LLM 스트림 수신 → 툴 실행 → 상태 갱신 → 루프 재진입 또는 종료"의 네 단계를 거친다.

Module 1과의 차이점: Module 1은 "어떤 일이 일어나는가"를 타임라인으로 설명했다. Module 2는 "왜 그렇게 구현됐는가" — 루프 상태, 분기 조건, 불변식, 안전장치의 설계 이유를 설명한다.

플로우차트 읽는 법: ABORT_STREAM 노드는 for await 스트림 루프가 완전히 끝난 뒤 중단 여부를 확인하는 지점이다(이터레이션 중간이 아님). 또한 StreamingToolExecutor는 스트림 수신 중에도 이미 도착한 tool_use 블록을 즉시 실행하기 시작하며, 중단 경로에서는 getRemainingResults()로 미완료 툴 결과를 드레인한다. 플로우차트의 단일 ABORT_STREAM 노드는 이 동시성을 가독성을 위해 압축한 것이다.

루프의 핵심 분기점은 두 곳이다.

  • needsFollowUp: 스트림에서 tool_use 블록이 하나라도 왔으면 true. 툴을 실행하고 다음 이터레이션으로 간다.
  • auto-continuation: 툴 콜이 없는데 LLM이 "이제 X를 하겠다"는 텍스트를 냈을 때, 루프가 자동으로 nudge 메시지를 주입하고 재진입한다.

핵심 메커니즘 해설

① 스트리밍 파싱: 이벤트를 쌓는 방식

deps.callModel()은 AsyncGenerator다. for await (const message of deps.callModel(...))로 이벤트를 하나씩 받는다. 각 이벤트 유형은 다르게 처리된다.

  • assistant 메시지: assistantMessages.push(message)로 누적. 이 배열이 나중에 API에 다시 전달된다.
  • tool_use 블록: toolUseBlocks.push(...)로 별도 수집. needsFollowUp = true 플래그 세팅.
  • stream_event: message_start에서 usage 리셋, message_delta에서 stop_reason 포착, message_stop에서 누적 usage 집계.
중요: stop_reason은 message_delta에서만 온다. 스트리밍 방식에서 assistant 메시지가 생성될 때는 아직 stop_reason === null이고, 이후 message_delta 이벤트가 실제 값을 채운다. queryLoop()가 추적하는 lastStopReason 클로저 변수가 이 값을 저장해 루프 종료 판정에 사용하는 이유다.

② 툴콜 누적과 실행

스트림이 끝나고 needsFollowUp === true면 runTools()(또는 StreamingToolExecutor)가 호출된다. 툴 결과는 toolResults에 쌓이고, 다음 이터레이션의 messages에 합쳐져 API로 전달된다.

StreamingToolExecutor는 스트리밍 도중 tool_use 블록이 도착하는 즉시 실행을 시작하는 최적화다. 모델이 스트리밍 중일 때도 이미 받은 툴은 병렬로 돌릴 수 있다.

③ 종료 조건: 주요 종료 경로

종료 유형조건반환 reason
자연 종료 툴 콜 없음 + auto-continuation 불필요 + stop hook 통과 completed
강제 중단 AbortController.abort() 호출 (Ctrl+C 등) aborted_streaming / aborted_tools
한도 초과 maxTurns 초과, 예산 초과, 컨텍스트 한계 max_turns / blocking_limit / ...
goal 달성 stop hook 평가 → preventContinuation = true stop_hook_prevented
오류 종료 API 에러, 이미지 크기, 툴 실패 루프 탐지 model_error / tool_failure_loop / ...

④ Auto-continuation: 왜 존재하고 왜 문제였나

LLM이 "이제 파일을 수정하겠습니다." 같은 텍스트를 출력하고 툴 콜 없이 멈추는 일이 생긴다. truncation(컨텍스트 잘림), 모델 버그, 또는 단순히 말을 먼저 한 후 행동하려는 패턴 때문이다. 이를 그냥 종료하면 작업이 미완성으로 끝난다.

auto-continuation은 이 상황을 감지해 nudge 메시지를 주입한다. analyzeContinuationIntent()가 마지막 텍스트의 마지막 120자를 분석해 "계속 의도"(action-verb 패턴, 콜론 종결, 미완 문법 구조 등)를 탐지하면 "Continue with the task..." 사용자 메시지를 몰래 끼워넣고 루프를 재진입시킨다.

비-Claude 모델에서 문제가 된 이유 (커밋 a5a8ccc 맥락): 원래 continuation 판정 로직은 Claude 모델의 응답 패턴에 맞게 튜닝되어 있었다. GPT나 Gemini 같은 비-Claude 모델은 문장 종결 방식이 다르다. 예를 들어 "All done." 처럼 명확히 완료했다고 써도 패턴 매처가 오탐해 불필요한 nudge를 계속 주입, 무한 루프에 가까운 상황이 벌어질 수 있었다. 커밋 a5a8ccc는 이 로직을 모델에 덜 편향된 방식으로 리팩터링해 completion marker(done, finished, completed 등) 감지를 명시적으로 강화했다. MAX_CONTINUATION_NUDGES(20)라는 하드 캡도 이 문제에 대한 안전망이다.

⑤ 상태 객체(State)의 역할

queryLoop()는 재귀 호출 대신 let state: State를 교체하는 방식으로 루프를 진행한다. continue 사이트마다 state = { ...nextState, transition: { reason: '...' } }를 쓴다. 이 패턴의 장점은 두 가지다.

  • 콜 스택을 소비하지 않아 N턴 무한 루프도 스택 오버플로 없이 돌 수 있다.
  • transition.reason 필드가 "왜 이 이터레이션이 시작됐는가"를 기록해 테스트에서 검증 가능하다.

실제 코드 예시

"직접 만든다면" — 견고한 루프의 불변식

에이전트 루프를 처음 만들면 대부분 간단한 while (needsTool)로 시작한다. 하지만 프로덕션 수준의 안정성을 얻으려면 다음 불변식들을 반드시 설계해야 한다.

불변식 1: messages 배열의 일관성

Anthropic API는 메시지 순서에 엄격하다. assistant 메시지 뒤에는 반드시 그 안의 모든 tool_use id에 대응하는 tool_result가 담긴 user 메시지가 와야 한다. 중단이 발생할 때도 yieldMissingToolResultBlocks()가 synthetic error tool_result를 채워 넣는 이유가 이것이다.

규칙: tool_use 블록을 messages에 push했으면 반드시 대응 tool_result를 push하고 나서 다음 API를 호출하라. 어느 경로로 종료하든 이 쌍이 깨져선 안 된다.

불변식 2: 무한 루프 방지 — 최소 두 가지 캡

루프에는 반드시 카운터 기반 탈출구가 두 개 이상 있어야 한다.

  • maxTurns: 이터레이션 횟수 상한. 툴 결과가 있어도 이 수에 도달하면 즉시 종료.
  • MAX_CONTINUATION_NUDGES (20): nudge 주입 횟수 상한. 모델이 툴도 안 쓰고 "계속할게요"만 반복하는 상황을 차단.
  • MAX_OUTPUT_TOKENS_RECOVERY_LIMIT (3): 출력 토큰 한계 도달 시 복구 재시도 상한.
카운터 하나만으로는 부족하다. 각 실패 모드별로 별도 카운터가 필요하다. openclaude는 maxOutputTokensRecoveryCount, continuationNudgeCount, hasAttemptedReactiveCompact(불리언 원샷 플래그) 등을 개별 필드로 관리한다.

불변식 3: 상태는 이터레이션 간에만 공유, 이터레이션 내부는 지역 변수

openclaude의 State 타입을 보면 이터레이션 시작 시 구조분해하고, 끝에서 새 객체를 만들어 state = next로 통째로 교체한다. 중간에 부분적으로 변형하지 않는다. 이 패턴 덕분에 각 이터레이션은 불변 스냅샷에서 시작하고, continue 사이트만이 다음 상태를 결정한다 — 버그가 숨기 어렵다.

불변식 4: 오류는 응답으로, 예외는 마지막 수단

API 에러(rate limit, prompt too long, max_output_tokens)는 throw되지 않고 synthetic assistant 메시지로 yield된다. 루프는 그 메시지를 감지해 복구를 시도하거나 gracefully 종료한다. JS 예외(throw)는 정말 예상 못 한 상황에만 사용한다. 이 원칙이 없으면 catch 블록 위치에 따라 messages 배열이 불일치 상태로 남는다.

불변식 5: 중단 신호는 어느 지점에서도 반응해야 한다

AbortController를 사용하고, 스트리밍 직후·툴 실행 직후·다음 이터레이션 진입 전 — 세 지점에서 확인한다. 신호가 들어오면 지금까지 쌓인 tool_use에 대해 synthetic result를 채워 넣고 종료한다. "다음 이터레이션에서 확인하겠다"는 태도는 응답성을 떨어뜨린다.
