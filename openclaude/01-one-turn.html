Module 1

한 턴의 전체 흐름

사용자가 엔터를 누른 순간부터 최종 텍스트가 출력될 때까지 — 코드 레벨의 타임라인

왜 이 개념인가

Module 0의 조감도에서 우리는 openclaude의 전체 아키텍처를 레이어별로 살펴봤다. 이제 그 지도 위에 실제 한 번의 대화가 어떻게 흘러가는지를 그려보자.

"엔터를 친다"는 이벤트는 생각보다 많은 일을 촉발한다. 입력 문자열이 UI를 떠나 API 호출이 되기까지, 그리고 LLM 응답이 스트림으로 돌아와 툴 호출을 감지하고 실행하고 결과를 다시 LLM에 보내는 왕복이 몇 번이나 일어나는지 — 이 과정을 모르면 디버깅도, 확장도 어렵다.

핵심 용어부터 정의하자.

  • 턴(turn): 사용자 입력 하나에서 시작해, 최종 result 메시지가 발행될 때까지의 전체 생명주기. QueryEngine.submitMessage() 한 번 호출과 같다.
  • 스텝(step / iteration): 쿼리 루프의 반복 한 번. LLM을 한 번 호출하고, 그 결과를 처리하는 단위. 툴 콜이 있으면 스텝이 추가로 발생한다.
한 턴 안에 스텝이 여러 번 발생할 수 있다. LLM이 툴을 부를 때마다 스텝이 하나 더 돈다. 사용자는 이를 인지하지 못하지만, 코드 내부에서는 while (true) 루프가 여러 번 반복된다.

핵심 메커니즘 해설

턴 vs 스텝 — 왜 구별이 중요한가

사용자 관점에서는 "한 번 물어보면 한 번 답이 온다"처럼 느껴진다. 하지만 코드 내부에서 QueryEngine.submitMessage()는 단 한 번만 호출되어도, 그 안의 queryLoop()는 while (true)를 돌며 LLM을 여러 번 부를 수 있다.

턴 카운터(turnCount)는 툴 결과(user 타입 메시지)가 도착할 때마다 증가한다. 즉 "LLM → 툴콜 → 결과 → LLM → 툴콜 → 결과 → LLM → 최종 텍스트"라면 turnCount는 3이 된다.

툴 콜이 왜 루프를 다시 도는가

LLM은 응답 안에 tool_use 타입의 콘텐츠 블록을 포함시킬 수 있다. 스트리밍 중 이 블록이 감지되면 needsFollowUp = true로 세트된다. 스트림이 끝나면 루프는 runTools()를 호출해 해당 툴을 실행하고, 결과를 tool_result 형태로 메시지 배열에 추가한 뒤 다음 스텝으로 continue한다.

LLM 입장에서 보면: "나는 툴을 호출하는 응답을 냈고, 그 결과가 다시 내 컨텍스트에 담겨서 돌아왔다. 이제 그 결과를 바탕으로 다시 응답한다." 이 왕복이 툴 호출 수만큼 반복된다.

언제 루프가 멈추는가

루프가 종료되는 조건은 다음과 같다 (정상 종료 기준):

  • needsFollowUp == false — LLM이 툴을 부르지 않고 텍스트만 반환했을 때
  • maxTurns 초과 → error_max_turns 결과
  • maxBudgetUsd 초과 → error_max_budget_usd 결과
  • 사용자 중단(abort) → aborted_streaming 또는 aborted_tools
  • 프롬프트 길이 초과(blocking limit) → 즉시 에러 반환

스트리밍과 yield*의 관계

queryLoop()는 AsyncGenerator다. LLM 스트림에서 오는 이벤트를 (stream_event), 완성된 어시스턴트 메시지(assistant), 툴 실행 결과(user)를 모두 yield하며 위로 전달한다. QueryEngine이 이 스트림을 for await로 소비하면서 SDK 규격에 맞는 메시지로 변환해 최종적으로 호출자에게 내보낸다.

실제 코드 예시

QueryEngine은 하나의 대화 세션을 소유한다. submitMessage()는 그 세션에서 새 턴을 시작하는 유일한 진입점이다. 반환형이 AsyncGenerator<SDKMessage>인 점에 주목하라 — 응답이 완성될 때까지 여러 이벤트가 yield되어 스트리밍 방식으로 전달된다.

QueryEngine은 query()(실체는 queryLoop())를 for await로 소비한다. 루프 안에서 오는 모든 메시지를 받아 SDK 규격으로 변환하거나 내부 상태(mutableMessages, 사용량 등)를 업데이트한다.

LLM 응답이 스트림으로 도착할 때, 어시스턴트 메시지 안의 tool_use 블록을 추출해 toolUseBlocks 배열에 쌓는다. needsFollowUp = true가 세트되는 순간 이 스텝 이후에 툴 실행 단계가 실행된다.

runTools()는 toolUseBlocks를 순회하며 각 툴을 실행한다. 결과는 user 타입의 tool_result 메시지로 변환되어 toolResults 배열에 쌓인다. 이 배열이 다음 스텝의 메시지 컨텍스트로 합쳐진다.

툴 실행이 끝나면 기존 메시지 배열(messagesForQuery)에 어시스턴트 응답(assistantMessages)과 툴 결과(toolResults)를 모두 합쳐 새 state를 만들고 while 루프 상단으로 돌아간다. LLM은 이제 툴 결과가 포함된 전체 대화 기록을 받아 다음 응답을 생성한다.

"직접 만든다면" 교훈

LLM 에이전트를 직접 구현한다면 핵심은 메시지 배열을 관리하는 while 루프다. openclaude의 queryLoop()가 하는 일을 최소한으로 추출하면 이렇게 된다:

핵심 인사이트 세 가지:

  1. LLM은 상태를 가지지 않는다. 매 호출마다 전체 대화 기록을 보내야 한다. "기억"은 LLM 안에 있는 게 아니라 messages 배열 안에 있다.
  2. 툴 결과는 user 역할 메시지다. Anthropic API 규격에서 툴 결과는 role: 'user'의 tool_result 블록으로 전달된다. 이것이 openclaude에서 turnCount가 user 메시지를 카운트하는 이유다.
  3. 루프 종료 조건을 반드시 설계하라. LLM이 툴을 계속 부르면 루프가 무한히 돌 수 있다. openclaude는 maxTurns, maxBudgetUsd, 툴 실패 루프 가드(toolFailureLoopGuard), 연속 컨티뉴에이션 한도를 둔다.
"에이전트 = LLM + 루프 + 도구 실행 + 상태(메시지 배열)" — 이 네 가지 외에 마법은 없다. openclaude는 이 단순한 구조 위에 스트리밍, 권한 검사, 컨텍스트 압축, 다중 모델 폴백 등을 정교하게 쌓은 것이다.
