Module 6

서브에이전트 & 오케스트레이션

하나의 에이전트가 여러 하위 에이전트를 낳는 방법 — 위임·병렬성·팀 구성

왜 이 개념인가

LLM에는 컨텍스트 창 한계가 있다. 거대한 코드베이스를 단 하나의 에이전트 루프에서 탐색·수정·검증하려 하면 프롬프트가 수십만 토큰을 넘어 포화 상태가 된다. 더 근본적인 문제도 있다. 한 루프는 직렬이다. 파일 A를 읽는 동안 파일 B를 읽을 수 없고, 리팩터링을 하는 동안 테스트를 돌릴 수 없다.

서브에이전트 오케스트레이션은 세 가지 이점을 제공한다.

이점 설명
컨텍스트 분할 각 서브에이전트는 자신의 좁은 문제에만 집중한다. 탐색 에이전트는 탐색만, 구현 에이전트는 구현만.
병렬성 독립적인 작업은 동시에 실행된다. 파일 A 분석과 파일 B 분석을 동시에 띄운다.
전문화 각 에이전트는 다른 시스템 프롬프트·모델·툴 집합을 받는다. Explore 에이전트는 읽기 전용 툴만, Worker 에이전트는 쓰기 툴까지.
핵심 통찰: 서브에이전트를 호출한다는 것은 또 하나의 QueryEngine 루프를 새로 시작하는 것이다. Module 2에서 배운 그 루프와 완전히 같은 구조다 — 시스템 프롬프트, 메시지 배열, query() 호출, 툴 실행 반복. 다만 이번에는 부모 루프의 툴 실행 안에서 자식 루프가 돌아간다.

코디네이터 모드에서의 팀 구조

CLAUDE_CODE_COORDINATOR_MODE=1 환경 변수를 설정하면 특별한 시스템 프롬프트가 주입되어 부모가 "코디네이터" 역할을 맡는다. 코디네이터는 워커만 스폰하고, 워커들은 subagent_type: "worker"로 지정된 전용 에이전트 정의를 사용한다.

핵심 메커니즘 해설

① 서브에이전트 = 또 하나의 QueryEngine 루프

runAgent() 함수(src/tools/AgentTool/runAgent.ts)의 핵심 루프는 for await (const message of query({...}))다. Module 2에서 배운 query()가 그대로 재사용된다. 다만 다음 세 가지가 달라진다.

  • 시스템 프롬프트: 에이전트 정의(frontmatter)에서 별도 프롬프트를 가져온다.
  • 툴 풀: 부모와 독립적으로 조립된다. 비동기 에이전트는 권한 프롬프트를 띄울 수 없으므로 shouldAvoidPermissionPrompts: true가 설정된다.
  • AbortController: 비동기 에이전트는 부모와 분리된 새 컨트롤러를 받는다. ESC를 눌러도 백그라운드 에이전트는 계속 실행된다.

② 동기 vs. 비동기 스폰

shouldRunAsync 플래그가 결정한다. run_in_background: true, isCoordinator, forceAsync 중 하나라도 참이면 비동기 경로로 간다.

  • 동기: AgentTool.call()이 블로킹된다. 서브에이전트가 끝날 때까지 부모 루프가 멈춘다.
  • 비동기: registerAsyncAgent()로 태스크를 등록하고 즉시 { status: "async_launched" }를 반환한다. 실제 실행은 void runAsyncAgentLifecycle(...) 형태로 분리된 프로미스로 시작된다.

비동기 에이전트가 완료되면 enqueueAgentNotification()이 부모의 메시지 큐에 <task-notification> XML을 삽입한다. 부모 루프는 이를 일반 사용자 메시지처럼 수신해 처리한다.

③ 태스크 등록·추적·중단

모든 비동기 에이전트는 LocalAgentTaskState(타입 공용체 TaskState의 일원)로 AppState.tasks에 등록된다. 상태는 pending → running → completed/failed/killed 순으로 전이된다. 사용자가 ESC를 누르거나 TaskStop 툴을 호출하면 killAsyncAgent()가 해당 에이전트의 AbortController.abort()를 호출한다.

④ 코디네이터 모드

CLAUDE_CODE_COORDINATOR_MODE=1 환경 변수가 설정되면 isCoordinatorMode()가 true를 반환하고, 두 가지 변화가 생긴다.

  • getCoordinatorSystemPrompt()가 주입된다 — "당신은 코디네이터입니다. 워커를 스폰하고, 결과를 합성하십시오."
  • 모든 AgentTool 스폰이 강제 비동기로 실행된다 (shouldRunAsync 내 isCoordinator 조건).

⑤ 팀(Team) 구성

TeamCreateTool은 팀 파일(team.json)을 디스크에 쓰고, AppState.teamContext를 업데이트한다. 이후 AgentTool에서 name 파라미터와 team_name이 함께 전달되면 인-프로세스 혹은 tmux 기반의 "teammate"를 스폰하는 경로(spawnTeammate())로 분기된다. 팀원들은 서로 독립적인 프로세스에서 실행되며 SendMessage 툴로 메시지를 주고받는다.

재귀 에이전트의 위험

에이전트가 에이전트를 낳을 수 있다면, 에이전트가 에이전트를 낳는 에이전트를 낳는 상황도 가능하다. 이것이 무한 분기(infinite branching) 문제다. 깊이 제한이 없으면 실행 비용은 기하급수적으로 불어난다.

openclaude의 가드: 포크 서브에이전트(FORK_SUBAGENT 피처 게이트) 경로에서는 자식이 부모의 대화 맥락을 복사해 또 다른 포크를 시도할 수 있다. 이를 차단하기 위해 두 겹의 가드가 있다. 첫 번째는 toolUseContext.options.querySource 확인 — 포크 자식임을 컨텍스트 수준에서 기록해 오토컴팩트(메시지 재작성)에도 살아남는다. 두 번째는 isInForkChild(messages) — 메시지 히스토리에서 보일러플레이트 태그를 스캔한다 (src/tools/AgentTool/forkSubagent.ts:81-92). 어느 쪽이든 포크 시도가 감지되면 throw new Error('Fork is not available inside a forked worker.')로 즉시 차단한다.

비용·복잡도 관리 원칙

원칙 이유 openclaude의 구현
깊이 제한 재귀 폭발 방지 포크 자식은 에이전트 스폰 불가 (querySource 가드)
백그라운드 비활성화 플래그 테스트·디버깅 시 예측 가능성 CLAUDE_CODE_DISABLE_BACKGROUND_TASKS=1
최대 동시 에이전트 수 과금 폭발 방지 Copilot 모드: GITHUB_COPILOT_MAX_SUBAGENTS 환경 변수
AbortController 분리 사용자 취소 시 부모만 중단, 백그라운드 유지 비동기 에이전트는 새 컨트롤러 생성

설계 교훈 요약

오케스트레이션은 강력하지만 에이전트 트리가 깊어질수록 디버깅이 지수적으로 어려워진다. openclaude가 보여주는 핵심 패턴은 평탄한 계층구조다 — 코디네이터 시스템 프롬프트에는 "팀 로스터는 평탄하다(flat), teammate는 다른 teammate를 스폰할 수 없다"고 명시되어 있다. 워커가 결과를 내면 코디네이터가 합성해 다음 명령을 내린다. 이 단순한 규칙 하나가 무한 재귀와 고비용 분기를 구조적으로 막는다.

직접 만든다면: 깊이 1 이상의 에이전트 재귀를 허용하기 전에 비용 상한·타임아웃·명시적 중단 신호를 반드시 구현하라. 그리고 코디네이터는 "생각하는 역할", 워커는 "실행하는 역할"로 분리하면 프롬프트 품질과 결과 예측 가능성이 크게 높아진다.
