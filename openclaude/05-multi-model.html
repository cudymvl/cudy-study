Module 5

멀티모델 추상화

하나의 에이전트 루프로 200+ 모델 지원 — 프로바이더 차이를 흡수하는 통합 레지스트리, 포트/어댑터 설계, XML 툴콜 복구 shim

왜 이 개념인가

openclaude의 에이전트 루프(Module 2)는 LLM을 호출할 때 항상 같은 코드 경로를 밟는다. 그러나 실제로는 OpenAI의 Chat Completions API, Anthropic의 Messages API, Google의 Gemini API, Ollama 같은 로컬 런타임 등 HTTP 규약도, 툴콜 포맷도, 인증 방식도 모두 다른 서비스들을 대상으로 한다.

이 문제를 "그냥 if/else로 분기한다"고 생각하면 순식간에 수백 줄의 스파게티가 된다. 모델 하나 추가할 때마다 루프 내부를 건드려야 하고, 새 프로바이더가 생기면 모든 분기를 다시 검토해야 한다.

openclaude가 택한 전략은 인테그레이션 추상화 계층(integration abstraction layer)이다.

  • 디스크립터(descriptor): 각 프로바이더·게이트웨이·모델의 특성을 선언적 객체로 표현한다. 코드가 아닌 데이터다.
  • 레지스트리(registry): 모든 디스크립터를 Map에 보관하고 지연 로딩(lazy loading)으로 시작 비용을 줄인다.
  • 어댑터(adapter/shim): 각 TransportKind마다 실제 HTTP 변환 로직을 담는다. 루프는 어댑터만 호출한다.
  • 호환성 브리지(compatibility bridge): 저장된 프로파일의 레거시 preset 이름을 현재 route id로 투명하게 변환한다.
핵심 통찰: 에이전트 루프는 어떤 모델을 쓰는지 모른다. 루프가 아는 것은 "이 route의 transport kind가 무엇인가"뿐이다. 나머지는 모두 어댑터가 처리한다.

덕분에 새 프로바이더 추가는 디스크립터 파일 하나를 작성하는 일로 줄어든다. 루프·툴 시스템·컨텍스트 관리 코드는 전혀 손댈 필요가 없다.

통합 메시지 흐름 — 프로바이더 어댑터 계층

에이전트 루프에서 API 응답이 돌아오기까지 메시지와 툴콜 포맷이 어떻게 변환되는지를 보여준다. JSON 구조 툴콜과 XML 텍스트 툴콜이 하나의 Anthropic 스트리밍 이벤트로 수렴하는 지점에 주목하라.

포맷 차이 변환 지점: openaiShim.ts는 스트리밍 중에 XML <tool_call> 오프너를 감지하면 해당 델타를 화면에 흘리지 않고 내부 버퍼(xmlToolCallText)에 축적한다. 스트림이 끝날 때(finalize 단계) parseXmlToolCalls()로 파싱하여 Anthropic tool_use 블록 이벤트로 변환한다. 루프는 이 차이를 전혀 인식하지 못한다.

핵심 메커니즘 해설

3.1 Integration Descriptor — 선언적 프로바이더 정의

각 프로바이더는 VendorDescriptor 또는 GatewayDescriptor 객체 하나로 표현된다. 이 객체는 코드를 담지 않는다. 오직 이 프로바이더가 어떤 것인지를 기술한다.

  • transportConfig.kind: 어떤 HTTP 어댑터를 쓸지 결정하는 핵심 필드. 'anthropic-native' | 'openai-compatible' | 'gemini-native' | 'bedrock' | 'vertex' | 'local'.
  • transportConfig.openaiShim: openai-compatible 계열에서 모델별 세부 조정값. 예: maxTokensField(일부 모델은 max_tokens, 다른 모델은 max_completion_tokens), thinkingRequestFormat(추론 모델의 chain-of-thought 직렬화 방식).
  • setup.authMode: 인증 방식. 'api-key' | 'oauth' | 'adc' | 'token' | 'none'.
  • catalog: 이 route가 지원하는 모델 목록. source: 'static' | 'dynamic' | 'hybrid'로 정적 목록과 동적 검색을 조합할 수 있다.
  • preset: UI/UX용 프리셋 메타데이터. 사용자 프로파일 저장 시 이 preset id를 사용한다.

3.2 Registry — 지연 로딩 Map

registry.ts는 다섯 개의 Map을 유지한다: _brands, _vendors, _gateways, _anthropicProxies, _models. 생성된 디스크립터 카탈로그는 약 10,000줄에 달하는 코드다. 이를 앱 시작 시점에 모두 평가하면 초기 로딩이 느려진다.

해결책: index.ts가 setRegistryLazyLoader(ensureIntegrationsLoaded)를 호출해 "첫 번째 조회 시 로드" 훅을 등록한다. 레지스트리를 읽는 모든 getter(getVendor, getModel 등)는 내부적으로 ensureLoaded()를 호출하고, 이 함수가 아직 로드되지 않은 경우에만 카탈로그를 평가한다. 재진입 가드(_lazyLoaderRunning)는 loader가 registry를 읽을 때 무한 재귀를 방지한다.

3.3 Compatibility Bridge — 레거시 preset 이름 매핑

사용자 프로파일에는 "openai", "deepseek" 같은 preset 문자열이 저장된다. 디스크립터 시스템이 도입되면서 내부 route id 체계가 바뀌었고, 하위 호환이 필요해졌다.

compatibility.ts는 생성된 PROVIDER_PRESET_MANIFEST로부터 PRESET_ROUTE_MAP: Map<ProviderPreset, ProviderPresetManifestEntry>를 만든다. profileResolver.ts의 resolveProfileRoute()는 이 맵을 첫 번째로 시도하고, 실패하면 vendor id → gateway id → unknown-fallback 순서로 폴백한다.

3.4 Capability / TransportOverride 매칭

모델 카탈로그 엔트리(ModelCatalogEntry)는 모델별 capabilities와 transportOverrides.openaiShim을 가질 수 있다. 예컨대 동일한 openai-compatible route 내에서도 특정 모델만 maxTokensField: 'max_completion_tokens'를 써야 하는 경우, route 전체 설정을 덮어쓰지 않고 해당 엔트리에만 오버라이드를 선언한다. runtimeMetadata.ts의 resolveOpenAIShimRuntimeContext()가 이 오버라이드를 병합한다.

3.5 openai-shim XML 툴콜 복구

OpenAI 호환 API를 경유하는 오픈소스 모델 중 일부(GLM, Qwen, DeepSeek 계열)는 structured tool_calls 필드 대신 어시스턴트 메시지 텍스트 안에 XML을 끼워 넣는다. 이 XML이 그대로 화면에 표시되면 에이전트가 툴을 호출하지 못한 채 멈춘다.

세 가지 방언(dialect)이 존재한다:

Dialect예시해석 방법
A — function/parameter 태그 <tool_call><function=read_file><parameter=path>/x</parameter></function></tool_call> XML_FUNCTION_NAME_RE + XML_PARAMETER_RE 정규식
B — GLM native (arg_key/arg_value) <tool_call>read_file<arg_key>path</arg_key><arg_value>/x</arg_value></tool_call> 첫 토큰을 함수명으로, XML_ARG_PAIR_RE로 인자 추출
C — Hermes JSON <tool_call>{"name":"read_file","arguments":{"path":"/x"}}</tool_call> 태그 내부를 JSON으로 파싱

스트리밍 중에는 <tool_call> 오프너가 SSE 델타 경계에서 잘릴 수 있다. trailingXmlOpenerPrefixLen()이 이를 감지하여 잘린 접두사를 xmlHoldback에 보관하고, 다음 델타와 합쳐 재판별한다. 구조화된 tool_calls가 도착하면 오탐(false positive)으로 판단하고 버퍼를 텍스트로 방출한다.

실제 코드 예시

"직접 만든다면" 교훈

포트/어댑터(Ports & Adapters) 패턴으로 프로바이더 추상화

openclaude의 설계는 헥사고날 아키텍처(Hexagonal Architecture)의 포트/어댑터 패턴과 정확히 일치한다.

  • 포트(Port): TransportKind — 에이전트 루프가 알고 있는 "무엇을 요청할 수 있는가"의 계약.
  • 어댑터(Adapter): openaiShim.ts, claude.ts, vertexClient.ts — 포트 계약을 실제 외부 API로 번역하는 구현체.
  • 도메인 코어: 에이전트 루프, 툴 시스템 — 포트만 바라보며 어댑터를 모른다.

의사코드로 표현하면:

새 프로바이더 추가 시 체크리스트

  1. 기존 transport kind로 커버되는가? 대부분의 새 API는 OpenAI Chat Completions를 채용하므로 kind: 'openai-compatible'으로 충분하다. 특수 인증(Azure AD, AWS SigV4 등)이 필요하면 기존 게이트웨이 디스크립터(azure-openai.ts, bedrock)를 참고한다.
  2. 디스크립터 파일 작성: src/integrations/vendors/ 또는 src/integrations/gateways/에 파일 하나 추가. defineVendor() / defineGateway()는 TypeScript가 누락 필드를 컴파일 오류로 알려 준다.
  3. openaiShim 조정값 확인: 모델이 max_completion_tokens를 쓰는가, max_tokens를 쓰는가? 추론 content를 reasoning_content 필드로 보내는가? 이를 transportOverrides.openaiShim에 선언한다.
  4. 레지스트리 자동 등록: 생성 파이프라인이 디스크립터를 integrationArtifacts.generated.js에 수집한다. index.ts의 loadIntegrationArtifacts()가 이 파일을 읽어 자동으로 등록한다. 별도 등록 코드는 필요 없다.
  5. XML 툴콜 방언 확인: 모델이 structured tool_calls 대신 XML을 텍스트로 내보내는가? openaiShim.ts의 기존 세 방언 파서가 커버하는지 확인한다. 새 포맷이라면 정규식 하나와 분기 조건을 추가한다.
설계 교훈: "코드가 아닌 데이터로 프로바이더를 표현하라." 어댑터 코드는 극히 드물게 바뀐다. 프로바이더는 자주 추가된다. 변화 빈도가 다른 두 관심사를 분리하면 — 디스크립터(데이터) vs. 어댑터(코드) — 신규 프로바이더 추가 비용이 0에 수렴한다.
"루프는 어댑터를 모른다. 어댑터는 루프를 모른다. 둘을 연결하는 것은 타입 하나, TransportKind다."
