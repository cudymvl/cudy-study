Module 7

확장성: MCP · Skills · 슬래시 커맨드

코어를 건드리지 않고 기능을 확장하는 세 가지 축 — 외부 툴·절차적 지식·사용자 단축

왜 이 개념인가

에이전트가 유용해지려면 능력(capability)이 늘어나야 한다. 그런데 능력을 늘리는 방법이 "소스코드를 고친다"뿐이라면 모든 사용자가 같은 단일 바이너리에 갇힌다. openclaude는 코어를 전혀 건드리지 않고 기능을 확장할 수 있는 세 가지 독립적인 축을 제공한다.

축 무엇을 확장하나 작성 언어 누가 주로 쓰나
MCP 서버 툴 레지스트리 — 모델이 호출할 수 있는 tool_use 블록 목록 임의(Python·Go·Node 등), JSON-RPC 표준 준수 외부 서비스 연동, 조직 내부 API 노출
Skills 슬래시 커맨드 레지스트리 — 프롬프트 패턴·절차적 지식 Markdown (YAML 프런트매터 + 지시문) 팀 관행, 반복 작업 절차화
슬래시 커맨드 (builtin) REPL UI 커맨드 — 사용자가 /foo로 직접 호출 TypeScript (Command 타입 구현) CLI 기능 추가, 내부 도구

세 축의 공통점은 하나다: 런타임 등록(runtime registration). 프로세스가 시작된 뒤, 또는 MCP 서버가 연결된 뒤에 레지스트리에 삽입되며 에이전트 루프나 REPL은 매 턴 레지스트리를 조회한다. 따라서 코어 루프 코드를 고칠 필요 없이 기능을 넣고 뺄 수 있다.

모듈 3과의 구분
모듈 3에서는 툴 추상화(Tool 인터페이스, ToolDef, 권한 체크 흐름)를 다뤘다. 이 모듈은 그 추상화를 확장 지점으로 활용하는 세 가지 메커니즘을 다룬다.

핵심: 레지스트리는 두 개다. Tool Registry(모델이 API로 호출하는 것)와 Command Registry(사용자 또는 SkillTool이 슬래시 커맨드로 호출하는 것). MCP 서버 하나가 두 레지스트리 모두에 기여할 수 있다 — 일반 MCP 툴은 Tool Registry에, skill:// 리소스는 Command Registry에 들어간다.

핵심 메커니즘 해설

① MCP 서버 연결과 툴 노출

MCP(Model Context Protocol)는 Anthropic이 정의한 JSON-RPC 2.0 기반 표준이다. CLI는 @modelcontextprotocol/sdk의 Client를 사용해 stdio · SSE · Streamable HTTP · WebSocket 네 가지 전송 방식으로 외부 서버에 연결한다.

연결이 맺어지면 tools/list RPC를 호출해 서버가 노출하는 툴 목록을 받아온다. 각 툴은 MCPTool이라는 베이스 객체를 스프레드한 뒤 이름과 스키마를 오버라이드하는 방식으로 Tool Registry에 삽입된다. 이름 형식은 mcp__{serverName}__{toolName}이다.

MCP 서버가 resources/list를 지원하고 skill:// URI 접두사를 가진 리소스를 노출하면, mcpSkills.ts가 이를 읽어 Markdown 프롬프트로 변환하고 Command Registry에 추가한다. 이로써 MCP 서버 하나가 툴도 제공하고 Skills도 제공할 수 있다.

승인(permission): 기본적으로 모든 MCP 툴 호출은 behavior: 'passthrough' — 즉 매번 사용자에게 물어본다. 사용자가 허용 규칙을 저장하면 alwaysAllowRules에 기록되어 이후 호출은 즉시 통과된다.

② Skill 발견·로딩·주입

Skill은 ~/.claude/skills/, 프로젝트 .claude/skills/, 플러그인, 또는 바이너리에 번들된 형태로 존재한다. loadSkillsDir.ts의 getSkillDirCommands()가 디렉터리를 스캔해 Markdown 파일마다 YAML 프런트매터를 파싱하고 createSkillCommand()로 Command 객체를 생성한다. 이 결과는 메모이즈되어 getCommands()를 통해 Command Registry에 합류한다.

모델이 Skill을 실행하고 싶을 때는 SkillTool이라는 특수 툴을 호출한다. SkillTool.call()은 커맨드 객체를 찾아 프롬프트 내용을 컨텍스트에 주입하고 (인라인 실행), 또는 context: 'fork'면 서브에이전트를 생성해 격리 실행한다. 사용자가 직접 /commit처럼 입력하면 processPromptSlashCommand()를 거쳐 같은 Command 객체가 처리된다.

번들 Skills: 디스크 없이 CLI 바이너리에 내장된 Skills는 registerBundledSkill()로 시작 시점에 등록된다. getBundledSkills()는 이 내부 배열을 반환한다.

③ 슬래시 커맨드 파싱·디스패치

builtin 슬래시 커맨드(예: /mcp, /config)는 commands.ts의 COMMANDS()에 정적으로 등록되는 TypeScript 객체다. 각 커맨드는 Command 타입을 만족하며 type: 'prompt'(프롬프트 확장) 또는 type: 'local'(UI 직접 조작) 중 하나다.

사용자 입력이 /로 시작하면 REPL이 findCommand()로 레지스트리를 검색한다. 일치하는 커맨드가 type: 'prompt'면 getPromptForCommand()를 호출해 최종 프롬프트 블록을 생성한 뒤 에이전트 루프에 주입한다. type: 'local'이면 UI 코드를 직접 실행해 모델을 호출하지 않는다.

플러그인 레지스트리 설계 원칙

openclaude의 세 확장 축은 모두 같은 설계 패턴을 공유한다. 직접 유사한 시스템을 만들 때 참고할 수 있는 원칙을 정리한다.

설계 결정 openclaude의 선택 대안과 트레이드오프
등록 시점 메모이즈된 함수 — 최초 호출 시 한 번 평가 즉시 평가(빠르지만 유연성 낮음) vs 매 턴 재스캔(유연하지만 느림)
네임스페이스 mcp__{server}__{tool} 형식으로 충돌 방지 단일 평면 네임스페이스는 간단하지만 플러그인 충돌 불가피
베이스 오버라이드 MCPTool 스프레드 후 필드 재정의 클래스 상속보다 구성이 테스트·교체에 유리

신뢰 경계 — 외부 MCP를 안전하게 다루기

외부 MCP 서버는 신뢰할 수 없는 외부 코드다. 서버가 노출하는 툴 설명(description)이나 Skill 프롬프트에는 악의적인 지시문이 포함될 수 있다(prompt injection). openclaude가 적용하는 방어 계층을 살펴보자.

방어 1 — hooks 강제 제거 (mcpSkills.ts)
MCP에서 불러온 Skill의 YAML 프런트매터에 hooks 항목이 있어도 hooks: undefined로 강제 초기화한다. hooks는 쉘 명령을 실행할 수 있으므로, 원격 Skill이 사용자 워크스페이스에서 임의 코드를 실행하는 경로를 차단한다.
방어 2 — allowed-tools 제거 (mcpSkills.ts)
MCP Skill의 allowed-tools도 빈 배열로 초기화한다. 로컬 Skill이라면 이 필드가 alwaysAllowRules에 기록되어 툴 호출이 자동 승인되지만, 원격 Skill이 동일하게 동작하면 서버가 Bash 같은 고위험 툴을 무승인 자동 실행시킬 수 있다.
방어 3 — passthrough 기본값 (client.ts)
MCP 툴은 기본적으로 behavior: 'passthrough'다. 즉 모델이 툴을 호출하려 할 때마다 사용자에게 물어본다. 사용자가 명시적으로 허용 규칙을 저장해야만 자동 통과된다.
방어 4 — 설명 길이 제한 (client.ts)
MAX_MCP_DESCRIPTION_LENGTH = 2048. OpenAPI 기반 서버처럼 15–60KB에 달하는 설명을 덤프하는 서버가 있다. 잘라내지 않으면 컨텍스트 창을 낭비하고, 극단적인 경우 간접 주입 공격의 표면이 된다.

설계 요약

외부 플러그인을 안전하게 통합하려면 "허용 목록 사고방식"이 필요하다. 플러그인이 할 수 있는 것의 최소 집합만 명시적으로 허용하고, 나머지는 기본적으로 차단한다. openclaude의 SAFE_SKILL_PROPERTIES 집합 (SkillTool.ts:885–918)이 그 좋은 예다: 미래에 새 프로퍼티가 추가되면 기본적으로 권한 요청이 필요하도록 설계되어 있다.
