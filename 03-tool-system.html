Module 3

툴 시스템

LLM의 유일한 외부 상호작용 통로 — 스키마 정의, 권한 게이트, 실행, 결과 직렬화, 지연 로딩의 모든 것

왜 이 개념인가

LLM은 본질적으로 텍스트 인/아웃 함수다. 파일을 읽거나 셸 명령을 실행하거나 웹을 검색하는 능력은 모델 가중치에 없다. 그 모든 능력은 툴(tool)을 통해서만 얻는다. Module 2에서 살펴본 에이전트 루프가 "언제 툴을 부르는가"의 이야기라면, 이번 모듈은 툴 그 자체 — 어떻게 정의되고, LLM에 어떻게 노출되며, 어떻게 실행·검증·결과화 되는가의 이야기다.

핵심 아이디어 하나를 먼저 잡자: 스키마(schema)가 곧 LLM의 API 문서다. 사람이 라이브러리 문서를 읽고 함수를 호출하듯, LLM은 각 툴의 JSON Schema를 읽고 인자를 조립한다. 스키마가 모호하면 LLM은 잘못된 인자를 만들고, 스키마가 정밀하면 LLM은 정확한 인자를 만든다. 이것이 툴 설계가 단순한 "인터페이스 정의"를 넘어 프롬프트 엔지니어링임을 뜻한다.

Module 2와의 구분: Module 2는 루프가 툴 콜을 발급하는 흐름을 다뤘다. 이 모듈은 툴 추상화 자체 — 인터페이스 구조, 검증, 권한, 결과 직렬화 — 를 다룬다.

툴 생명주기 플로우차트

하나의 툴 콜이 LLM 응답에서 출발해 결과로서 모델에 재투입되기까지의 전 과정이다. 각 단계는 독립된 책임을 가지며, 어느 단계에서든 거부·에러가 발생하면 그 사실 자체가 모델로 돌아간다. 

flowchart TD
    A["툴 정의\n(name · inputSchema · call · checkPermissions)"] -->|buildTool| B["레지스트리 등록\ngetAllBaseTools()"]
    B -->|assembleToolPool| C["툴 풀 조립\n(빌트인 + MCP, 알파벳 정렬)"]
    C -->|isToolSearchEnabled| D{지연 로딩?}
    D -->|아니오| E["전체 스키마 노출\n(Anthropic API tools[])"]
    D -->|예| F["defer_loading: true\n일부 스키마만 노출"]
    F -->|LLM이 ToolSearch 호출| G["ToolSearch\n키워드 매칭 → tool_reference 반환"]
    G --> E
    E -->|LLM 응답| H["tool_use 블록 수신\n{name, input, id}"]
    H --> I["validateInput\n(입력 의미 검증)"]
    I -->|실패| J["tool_result: error\n모델에 재투입"]
    I -->|통과| K["checkPermissions\n(파일경로·명령·모드 검사)"]
    K -->|거부| L["사용자 확인 요청\n또는 자동 거부"]
    L --> J
    K -->|허용| M["call() 실행\n(실제 I/O 수행)"]
    M --> N["ToolResult 반환\n{data, newMessages?}"]
    N --> O["mapToolResultToToolResultBlockParam\n(Output → ToolResultBlockParam)"]
    O --> P["tool_result 블록\n대화 컨텍스트에 삽입"]
    P -->|다음 API 호출| Q["LLM 재투입\n다음 추론 시작"]

핵심 메커니즘 해설

3-1. Tool 인터페이스 구조

src/Tool.ts에 정의된 Tool<Input, Output, P> 타입은 모든 툴이 반드시 구현해야 하는 계약이다. 핵심 멤버 다섯 가지를 먼저 이해하자.

멤버역할LLM과의 관계
name 툴 식별자 문자열 LLM이 tool_use.name으로 지정하는 값
inputSchema Zod 스키마 (JSON Schema로 직렬화) LLM의 "API 문서" — 인자 타입·설명·제약
call() 실제 I/O 수행 함수 LLM은 이 함수 존재를 모름; 결과만 받음
checkPermissions() 권한 게이트 — 허용/거부/사용자 확인 LLM에 투명; 거부 시 error tool_result
mapToolResultToToolResultBlockParam() Output → Anthropic SDK ToolResultBlockParam LLM이 받는 직렬화 형식

나머지 멤버들(validateInput, isReadOnly, isConcurrencySafe, renderToolResultMessage 등)은 검증·UI·분류 보조 역할이다. shouldDefer/alwaysLoad 플래그는 지연 로딩 전략에서 다룬다.

3-2. buildTool과 기본값 채우기

모든 툴은 직접 Tool 타입을 구현하지 않고 buildTool(def) 헬퍼를 거친다. 이 함수는 { ...TOOL_DEFAULTS, ...def }를 수행해 자주 생략되는 메서드에 실패 안전(fail-closed) 기본값을 채운다.

기본값 철학: isConcurrencySafe → false (동시 실행 불가 가정), isReadOnly → false (쓰기 가능 가정), checkPermissions → allow (범용 권한 시스템에 위임). 보안에 영향을 주는 방향으로는 항상 더 제한적인 방향을 기본으로 삼는다.

3-3. 입력 검증 — LLM 바깥에서 해야 하는 이유

LLM이 생성한 인자는 스키마에 맞더라도 의미가 잘못될 수 있다 (존재하지 않는 파일 경로, 막힌 sleep 패턴, UNC 경로 보안 우회 등). validateInput()은 이를 잡아내 에러 메시지를 모델에 돌려주는 두 번째 방어선이다.

검증을 LLM 안에서 하지 않는 이유는 단순하다: LLM은 동일 입력에 동일한 오류를 반복할 수 있고, 부작용 없는 검증이 불가하다. 그러나 코드 검증은 결정론적이며 부작용이 없다. 신뢰 경계(trust boundary)는 LLM 출력과 시스템 경계 사이에 있어야 한다.

3-4. 권한 게이트 — checkPermissions

validateInput이 통과한 후에만 checkPermissions가 호출된다. 반환 타입 PermissionResult는 세 가지 행동 중 하나다: 'allow'(즉시 실행), 'ask'(사용자 확인), 'reject'(거부). 각 툴은 자체 로직을 구현할 수 있고(BashTool의 명령어 패턴 매칭, GrepTool의 파일 읽기 경로 확인), 생략하면 범용 권한 시스템이 처리한다.

3-5. 결과 직렬화 — mapToolResultToToolResultBlockParam

call()이 반환하는 ToolResult<Output>은 TypeScript 타입이다. Anthropic API에 보내려면 ToolResultBlockParam 형식(type: 'tool_result', tool_use_id, content) 으로 변환해야 한다. 이 변환이 mapToolResultToToolResultBlockParam이다.

툴마다 변환 방식이 다르다: BashTool은 stdout·stderr를 문자열로 이어붙이고, GrepTool은 모드(content/files_with_matches/count)에 따라 다른 포맷을 만든다. ToolSearchTool은 tool_reference 블록 배열을 반환하는 특수 포맷을 사용한다.

3-6. ToolSearch와 지연 로딩 — 컨텍스트 절약 전략

툴이 많아질수록 시스템 프롬프트의 툴 스키마 섹션이 수천 토큰을 차지한다. MCP 서버를 여러 개 연결하면 수십~수백 개의 툴이 등록될 수 있다.

해법은 지연 로딩(deferred loading)이다. shouldDefer: true인 툴은 API 호출 시 defer_loading: true 플래그와 함께 스키마 없이 이름만 전달된다. LLM이 그 툴을 쓰고 싶으면 먼저 ToolSearch를 호출해 풀 스키마를 받은 후 사용한다. 이 추가 왕복(round-trip)은 컨텍스트 비용을 크게 줄인다.

반대로 alwaysLoad: true인 툴은 지연 없이 첫 번째 프롬프트부터 스키마가 노출된다. 첫 턴에 반드시 사용해야 하는 툴(예: ToolSearch 자체)에 쓴다.

왜 스키마 검증을 LLM 밖에서 하나

LLM은 함수 시그니처를 지키는 코드 컴파일러가 아니다. 같은 오류 입력을 받으면 같은 잘못된 호출을 반복할 수 있고, JSON 구문은 맞아도 의미가 틀린 인자를 생성할 수 있다. 반면 Zod 검증은 결정론적이고 부작용이 없으며 즉각적이다.

신뢰 경계 원칙: LLM 출력은 "신뢰하되 검증하라(trust but verify)"가 아니라 "항상 검증하라"다. 스키마 검증, 경로 이탈 검사, 명령어 패턴 매칭이 모두 이 원칙의 구현체다.

지연 로딩을 도입하는 시점

툴 스키마는 모두 시스템 프롬프트(또는 API tools[])에 포함되어 모든 API 호출의 컨텍스트를 소비한다. 툴 10개 × 평균 200토큰 = 2,000 토큰이 항상 낭비된다. 이것이 ToolSearch 방식의 지연 로딩이 필요한 이유다. 단, 지연 로딩은 추가 왕복(LLM → ToolSearch → LLM)을 유발하므로 자주 쓰이는 핵심 툴은 alwaysLoad, 드물게 쓰이는 툴은 shouldDefer로 나누는 것이 실용적이다.

openclaude의 또 다른 설계 포인트: 알파벳 정렬로 프롬프트 캐시를 안정화한다. assembleToolPool이 빌트인과 MCP 툴을 각각 정렬 후 합치는 이유는, MCP 서버 연결 순서가 달라져도 캐시 키가 바뀌지 않게 하기 위해서다. LLM 비용에서 캐시 히트율은 직접적인 비용 절감으로 이어진다.
