Module 8

권한 & 안전장치

자율 에이전트가 파일·셸에 접근할 때 신뢰를 유지하는 방법 — 권한 게이트의 설계와 구현

왜 이 개념인가

openclaude는 자율적(autonomous)으로 동작한다. 사용자가 자리를 비운 사이에도 파일을 편집하고, 셸 명령을 실행하고, 인터넷에 접근한다. 이 능력은 편리하지만 동시에 위험하다.

예를 들어 에이전트가 rm -rf /를 실행하거나, .ssh/id_rsa를 외부로 전송하거나, 프로젝트 설정 파일인 .claude/settings.json을 몰래 수정한다면 어떻게 될까? LLM은 환각(hallucination)을 일으킬 수 있고, 악의적인 프롬프트 인젝션에 속을 수도 있다.

핵심 원칙: 에이전트가 어떤 툴을 호출하기 직전에, "이 작업을 지금 이 맥락에서 허용해도 되는가?"를 묻는 권한 게이트(permission gate)가 반드시 개입한다. 이 게이트가 신뢰의 핵심이다.

openclaude의 권한 시스템은 세 가지 질문에 답한다.

  1. 무엇이 허용되는가? — 허용 규칙(allow rules)과 거부 규칙(deny rules)의 집합
  2. 지금 어떤 모드인가? — plan / acceptEdits / bypassPermissions / default 등
  3. 사용자가 이 순간 직접 승인했는가? — 터미널 다이얼로그 또는 훅(hook)

세 질문 중 하나라도 "허용"을 반환하지 않으면 툴 실행은 차단된다. 이 모듈에서는 그 판정이 어떻게 이루어지는지 코드 수준에서 추적한다.

핵심 메커니즘 해설

권한 타입 모델

src/types/permissions.ts는 순수 타입 정의만 담은 파일이다. 임포트 사이클을 끊기 위해 의도적으로 런타임 의존성을 제거했다.

주요 타입 계층은 다음과 같다.

타입역할
PermissionMode 'default' | 'plan' | 'acceptEdits' | 'bypassPermissions' | 'fullAccess' | 'dontAsk' | 'auto' | 'bubble' — 현재 세션의 전역 모드
PermissionBehavior 'allow' | 'deny' | 'ask' — 단일 규칙 또는 결정의 결과값
PermissionRule 출처(source) + 행동(ruleBehavior) + 대상(ruleValue)
PermissionDecision allow | ask | deny 중 하나. allow에는 updatedInput이 포함될 수 있어 입력값 변환도 허용
PermissionResult PermissionDecision + 'passthrough' (툴이 "나는 모르겠으니 상위로 넘김")
ToolPermissionContext 세션 전체에서 공유되는 불변 스냅샷: 현재 모드, allow/deny/ask 규칙 집합, bypass 가용 여부 등

Plan 모드 — 읽기전용 강제

EnterPlanMode 툴이 호출되면 setAppState로 toolPermissionContext.mode를 'plan'으로 전환한다. 이 상태에서는 쓰기·실행 계열 툴(FileEditTool, BashTool 등)의 checkPermissions()가 직접 behavior: 'ask'를 반환하여 사용자 승인 없이는 실행이 불가능하다.

ExitPlanMode 툴은 반드시 사용자 승인 다이얼로그를 거쳐야 실행된다 (requiresUserInteraction() { return true }). 이 조건은 판정 파이프라인 1e 단계에서 확인되어 bypass 모드에서도 무시되지 않는다.

bypassPermissions 모드 — 위험과 가드

bypassPermissions는 "모든 권한 프롬프트를 건너뜀"을 의미한다. CI/CD 파이프라인이나 완전히 격리된 컨테이너에서 사용하는 모드다. 그러나 무조건적인 우회가 아니다. 다음 상황에서는 bypass가 작동하지 않는다.

  • deny 규칙 (1a 단계): 명시적으로 거부된 툴은 bypass에서도 차단된다.
  • content-specific ask 규칙 (1f 단계): 예를 들어 Bash(npm publish:*)처럼 사용자가 명시적으로 "이 내용은 항상 물어봐"라고 설정한 규칙은 bypass에서도 묻는다.
  • safetyCheck 경로 (1g 단계): .git/, .claude/, .ssh/, 셸 설정 파일 등 민감한 경로는 bypass에서도 반드시 승인을 요구한다.
  • requiresUserInteraction 툴 (1e 단계): ExitPlanMode처럼 본질적으로 사용자 상호작용이 필요한 툴은 bypass에서도 우회되지 않는다.

또한 bypassPermissions나 fullAccess 모드로 처음 진입할 때는 BypassPermissionsModeDialog가 표시된다. 사용자가 이 경고를 명시적으로 수락해야만 모드가 활성화되고, 수락 여부는 userSettings에 skipDangerousModePermissionPrompt로 기록된다 (다음에는 재표시 안 함).

나아가 Statsig 기능 게이트(bypassPermissionsKillswitch.ts)가 원격에서 이 모드를 강제로 비활성화할 수 있다. 조직 레벨의 긴급 차단 스위치다.

툴별 권한 훅 — PermissionContext

각 툴 실행에는 PermissionContext 객체가 생성된다 (src/hooks/toolPermission/PermissionContext.ts). 이 객체는 권한 결정의 모든 생애주기를 담당한다.

  • runHooks(): PreToolUse 훅을 실행하여 외부 스크립트가 allow/deny 결정을 내릴 기회를 준다.
  • handleUserAllow() / buildDeny(): 사용자 응답을 PermissionDecision으로 변환하고 권한 업데이트를 영속화한다.
  • persistPermissions(): 사용자가 "항상 허용"을 선택했을 때 해당 규칙을 설정 파일에 저장한다.
  • cancelAndAbort(): 사용자가 거부 시 AbortController로 에이전트 루프 전체를 중단시킬 수 있다.

실제 코드 예시

"직접 만든다면" 교훈

1. Deny-by-default 설계 원칙

openclaude의 판정 파이프라인은 명시적으로 허용되지 않은 것은 모두 묻는다는 원칙으로 설계되었다. passthrough라는 특수 행동값이 이를 표현한다. 툴이 "잘 모르겠다"고 말하면 (passthrough), 파이프라인은 이를 'ask'로 격상시킨다. 모르면 막는다는 것이 기본값이다.

"규칙이 없으면 허용" 방식은 편리하지만, 규칙 하나를 빠뜨리는 순간 보안 구멍이 생긴다. 에이전트처럼 자율적인 시스템은 반드시 "규칙이 없으면 거부(deny) 또는 질문(ask)"으로 설계해야 한다.

2. 위험 작업 분류 기준

openclaude가 특정 경로와 작업을 "위험"으로 분류하는 기준은 다음과 같이 계층화되어 있다.

위험도 기준 bypass 반응
절대 거부 사용자 명시적 deny 규칙 bypass 무시, 항상 차단
bypass 면역 .git/, .claude/, .ssh/, .bashrc 등 민감 경로 (safetyCheck) bypass에서도 반드시 승인 요구
content-specific ask Bash(npm publish:*)처럼 특정 명령 패턴 bypass에서도 묻는다
일반 쓰기 작업 파일 편집, 셸 명령 실행 bypassPermissions에서 자동 통과
읽기 전용 작업 파일 읽기, 검색 기본적으로 항상 통과

3. 모드를 계층 구조로 설계하라

단순히 "on/off" 스위치가 아니라, 모드를 덜 제한적인 것에서 더 제한적인 것으로 계층화하면 사용자가 작업 맥락에 따라 적절한 수준을 선택할 수 있다. openclaude는 plan(읽기전용 탐색) → default(선택적 승인) → acceptEdits(파일 편집 자동 승인) → bypassPermissions(전체 자동 승인) → fullAccess(safetyCheck까지 우회) 순으로 단계가 있다. 각 단계는 이전 단계의 제약을 유지하거나 완화하는 방향으로만 이동한다.

4. 원격 킬스위치를 만들어라

bypassPermissionsKillswitch.ts의 패턴처럼, 위험한 모드는 서버 사이드 기능 게이트(feature gate)로 언제든지 강제 비활성화할 수 있어야 한다. 클라이언트에 배포된 코드는 회수가 불가능하지만, 기능 게이트는 즉시 변경할 수 있다. 이는 보안 취약점 발견 시 긴급 대응의 핵심 수단이다.

한 줄 요약: 권한 시스템은 기능이 아니라 아키텍처다. 에이전트가 코드를 작성하기 전에, 권한 게이트가 어디에 위치하고, 무엇이 그것을 우회할 수 있으며, 무엇이 절대로 우회되어서는 안 되는지를 먼저 설계해야 한다.
