Module 6

서브에이전트 주도 개발과 병렬화

에이전트가 몇 시간 동안 자율로 일하는 비결 — 컨텍스트 한계를 위임(delegation)으로 넘고, 독립 작업을 동시에 실행하고, worktree로 충돌을 원천 차단한다

왜 이 개념인가

AI 에이전트가 "몇 시간 자율로 일한다"는 말을 들으면 두 가지 의문이 생긴다: 어떻게 컨텍스트가 부족하지 않을까? 어떻게 여러 작업이 서로 꼬이지 않을까? Superpowers의 세 스킬 — subagent-driven-development, dispatching-parallel-agents, using-git-worktrees — 은 이 두 문제에 대한 구조적 답이다.

컨텍스트 한계 문제

LLM 세션에는 컨텍스트 창(context window)이 있다. 한 에이전트가 계획 전체를 품고 태스크 1부터 태스크 N까지 처리하면, 중반쯤 되면 초반 결정을 잊거나, 누적된 히스토리가 새 태스크에 간섭한다. 실제 세션에서 컨트롤러 에이전트가 누적 히스토리를 다음 서브에이전트 프롬프트에 그대로 붙여 넣은 결과 42,000자 중 99%가 이전 태스크 요약이 된 사례가 기록되어 있다. (subagent-driven-development/SKILL.md:191)

해법은 위임(delegation)이다. 오케스트레이터는 계획을 읽고 할 일 목록을 만든 뒤, 태스크마다 새 서브에이전트를 띄워 딱 그 태스크에 필요한 맥락만 전달한다. 서브에이전트는 세션 히스토리를 전혀 모른 채 깨끗한 컨텍스트로 시작한다. 오케스트레이터는 협조(coordination) 역할만 하므로 자신의 컨텍스트도 보존된다.

병렬 실행 문제

독립적인 태스크가 여럿 있을 때 순차 실행은 낭비다. 파일 A 버그, 파일 B 테스트 실패, 파일 C 기능 — 세 작업이 서로 무관하다면 동시에 처리할 수 있다. 문제는 에이전트들이 같은 파일을 동시에 고치면 충돌이 난다는 것. dispatching-parallel-agents는 독립성 조건을 명시하고, using-git-worktrees는 각 에이전트에게 격리된 파일 시스템 공간을 제공한다.

이 학습 사이트 자체가 증거다

이 Superpowers 학습 사이트는 subagent-driven-development를 써서 만들어졌다. 각 모듈마다 별도의 implementer 서브에이전트가 작성하고, task reviewer 서브에이전트가 검토했다. 오케스트레이터는 10개 모듈 전체 계획을 품고 있으면서도 컨텍스트를 아끼며 진행한다 — 바로 이 모듈이 설명하는 패턴의 라이브 사례다.

SDD 레인의 핵심은 게이트(gate)다: 각 태스크 사이에 reviewer가 있어 spec 준수와 코드 품질 두 가지를 모두 통과해야만 다음 태스크로 넘어간다. 병렬 레인의 핵심은 한 응답에 동시 dispatch다: 같은 응답에 여러 서브에이전트 호출을 내면 병렬 실행되고, 한 번에 하나씩 내면 순차가 된다.

핵심 메커니즘 해설

1. Subagent-Driven Development — fresh context per task

SDD의 핵심 전제는 이것이다: 서브에이전트는 오케스트레이터의 세션 히스토리를 절대 상속받지 않는다. 오케스트레이터가 직접 구성한 것만 — brief 파일 경로, 인터페이스 결정, 글로벌 제약 — 전달된다. 이 구조가 긴 세션에서도 각 태스크의 품질을 유지하게 한다.

두 단계 리뷰: 구현 후 task reviewer 서브에이전트가 두 가지를 판정한다: spec compliance(명세 초과/미달 없음)와 code quality(코드 품질). 두 판정 중 하나라도 실패하면 fix 서브에이전트를 dispatch하고 재검토한다. implementer의 자체 리뷰는 이 공식 리뷰를 대체하지 않는다.

Durable Progress (지속 가능한 진행): 컨텍스트 압축(compaction) 후 오케스트레이터가 위치를 잃으면 완료된 태스크를 재dispatch하는 가장 비싼 실패가 발생한다. 이를 막기 위해 진행 상황을 .superpowers/sdd/progress.md ledger 파일에 기록하고, 재시작 시 ledger를 먼저 읽어 완료 태스크를 건너뛴다.

모델 선택: 비용과 속도를 위해 태스크 복잡도에 맞는 최소 모델을 쓴다. 1-2 파일의 명확한 스펙은 저가 모델, 통합·판단이 필요한 작업은 중간 모델, 아키텍처 결정과 최종 branch 리뷰는 최고 모델. 모델을 명시하지 않으면 세션 기본값(대개 가장 비싼 모델)을 상속하여 비용이 증가한다.

2. Dispatching Parallel Agents — 독립성 조건이 전부다

병렬 dispatch는 독립성(independence) 판정이 선행되어야 한다. 다음 조건이 모두 성립할 때만 병렬이 유효하다:

  • 각 문제가 다른 루트 원인을 가진다 (수정 하나가 다른 것을 고치지 않는다)
  • 에이전트들이 같은 파일을 편집하지 않는다 (공유 상태 없음)
  • 한 결과를 알아야 다른 작업을 시작할 수 있지 않다 (순서 의존 없음)

조건이 성립하면 기술적으로 간단하다: 한 응답에 여러 서브에이전트 호출을 동시에 내면 병렬 실행된다. 한 번에 하나씩 내면 순차다. 도구 호출 타이밍이 병렬성을 결정한다.

각 에이전트 프롬프트는 네 가지를 갖춰야 한다: 명확한 스코프(하나의 파일/서브시스템), 문제 이해에 필요한 완전한 맥락, 범위 제한("다른 코드 변경 금지"), 기대 출력 형식. 범위가 넓거나("모든 테스트 고쳐") 맥락이 없으면 에이전트가 길을 잃는다.

3. Using Git Worktrees — 충돌 원천 차단

Worktree는 같은 git 저장소의 브랜치를 별도 디렉토리에서 동시에 체크아웃하는 기능이다. 에이전트 A가 .worktrees/feature-a/에서, 에이전트 B가 .worktrees/feature-b/에서 작업하면 파일 시스템 충돌이 물리적으로 불가능하다.

스킬의 4단계 프로세스:

  1. Step 0 — 기존 격리 감지: 이미 linked worktree 안에 있다면 새로 만들지 않는다. 서브모듈과의 혼동을 방지하는 guard 명령도 포함된다.
  2. Step 1 — 워크스페이스 생성: 플랫폼 네이티브 도구(EnterWorktree 등)를 먼저 시도하고, 없을 때만 git worktree add로 폴백한다.
  3. Step 2 — 프로젝트 셋업: package.json, Cargo.toml 등을 자동 감지해 의존성을 설치한다.
  4. Step 3 — 클린 베이스라인 검증: 새 버그와 기존 버그를 구분할 수 있도록 작업 전 테스트가 모두 통과하는지 확인한다. 실패하면 진행 여부를 묻는다.

중요한 안전 규칙: 프로젝트 내부 디렉토리(.worktrees/)를 쓸 때는 git check-ignore로 gitignore 등록 여부를 검증해야 한다. 미등록 시 worktree 내용이 저장소에 추적되는 오염이 발생한다.

실제 발췌

"직접 만든다면" — fresh context + 명확한 brief + 게이트 리뷰

이 세 스킬의 조합에서 배울 수 있는 가장 실용적인 교훈은 하나다: 긴 자율 작업은 컨텍스트 분리 + 게이트 리뷰의 산물이다. 오케스트레이터가 똑똑해서 몇 시간을 일하는 게 아니라, 구조가 각 서브에이전트를 좁고 명확한 문제에 집중시키기 때문에 전체 작업이 완수된다.

Fresh Context — 서브에이전트에게 넘기는 것과 넘기지 않는 것

SDD에서 배운 핵심 기술은 선택적 컨텍스트 전달이다. 서브에이전트에게 전달해야 할 것:

  • 이 태스크의 brief 파일 경로 (요구사항의 단일 출처)
  • 이전 태스크의 인터페이스 결정 (API 시그니처, 데이터 구조)
  • 글로벌 제약 (코딩 컨벤션, 금지 패턴)
  • 보고서 파일 경로 (서브에이전트가 여기에 쓴다)

전달하면 안 되는 것: 이전 태스크의 구현 요약, 세션 전체 히스토리, 미래 태스크 정보. 이것들은 서브에이전트의 컨텍스트를 낭비하고 간섭을 만든다.

명확한 Brief — 추측을 없애는 단일 출처

각 태스크의 brief는 "요구사항의 단일 출처"여야 한다. 디스패치 프롬프트에 exact value(숫자, 문자열, 함수 시그니처)를 인라인으로 쓰지 않는다 — 모두 brief 파일에 있고, 프롬프트는 brief 파일 경로만 가리킨다. 이렇게 하면 오케스트레이터 컨텍스트에 artifact가 쌓이지 않고, 재dispatch 시에도 같은 brief를 재사용할 수 있다.

Brief가 모호하면 서브에이전트가 질문한다 — 이것은 정상이고 환영해야 한다. 구현이 시작된 후 발견되는 모호함보다 시작 전에 해소하는 것이 훨씬 싸다.

게이트 리뷰 — "close enough"는 없다

자율 작업이 품질을 유지하는 비결은 각 태스크 뒤에 붙는 게이트 리뷰다. Spec compliance와 code quality 두 판정이 모두 통과해야 한다. 하나라도 실패하면 fix 서브에이전트를 보내고 재검토한다. "대체로 맞다" 또는 "minor한 문제라 넘어가자"는 허용되지 않는다.

이 규율이 불편하게 느껴질 수 있다. 하지만 태스크 10에서 발견한 태스크 3의 버그는 태스크 4~9를 재작업해야 할 수 있다. 게이트 리뷰는 비용이 아니라 보험이다.

병렬화의 적용 범위 — SDD와의 구분

중요한 구분이 있다: SDD는 구현 서브에이전트를 병렬로 dispatch하는 것을 명시적으로 금지한다("Dispatch multiple implementation subagents in parallel (conflicts)"). SDD의 순차 게이트 방식과 병렬 에이전트 dispatch는 용도가 다르다:

  • SDD: 계획된 태스크 시퀀스를 순서대로 실행하되, 각 태스크를 새 서브에이전트에 위임하여 컨텍스트를 격리한다. 구현 서브에이전트는 하나씩 실행된다.
  • Dispatching Parallel Agents: 독립적인 버그·기능·조사 작업이 여러 개 있을 때 동시 실행한다. 같은 코드를 고치는 구현 작업에는 쓰지 않는다.

두 스킬은 경쟁하지 않는다 — 각자 다른 문제를 푸는 도구다. worktree는 두 경우 모두에서 격리를 제공하는 기반 인프라다.

핵심 교훈: 에이전트가 자율적으로 오래 일할 수 있는 것은 그 에이전트가 특별히 똑똑해서가 아니다. fresh context로 시작하는 서브에이전트, 명확한 brief로 범위가 제한된 태스크, 게이트 리뷰로 품질이 보장된 각 단계 — 이 구조가 긴 자율 작업을 가능하게 한다. 구조 없이 하나의 에이전트에게 "다 해"라고 시키면 중간쯤에서 길을 잃는 것이 기본값이다.
