Module 5

모든 것에 TDD

Red-Green-Refactor는 코드에만 적용되지 않는다 — Superpowers는 스킬 문서 작성에도 같은 사이클을 강제하며, 이것이 시스템 전체의 척추가 된다

왜 이 개념인가

Superpowers를 처음 접하면 "TDD를 강요하는 또 다른 스킬 파일"로 볼 수 있다. 하지만 그것은 절반만 맞는 이해다. Superpowers에서 TDD는 두 개의 층위에서 작동한다.

첫 번째 층위 — 코드 TDD: 에이전트가 기능을 구현할 때 반드시 테스트를 먼저 작성하고, 실패를 확인하고, 최소한의 코드로 통과시키는 사이클. 이것은 TDD의 전통적인 의미이며, test-driven-development/SKILL.md가 이를 강제한다.

두 번째 층위 — 스킬 TDD: 이것이 Superpowers의 메타적 특징이다. 스킬 문서를 작성할 때도 RED-GREEN-REFACTOR를 적용한다. "테스트 케이스"는 서브에이전트로 실행하는 압박 시나리오(pressure scenario)이고, "프로덕션 코드"는 SKILL.md 문서 자체다. writing-skills/SKILL.md는 이를 명시한다:

"Writing skills IS Test-Driven Development applied to process documentation." — skills/writing-skills/SKILL.md:10

그리고 가장 핵심적인 원칙:

"에이전트가 스킬 없이 실패하는 것을 보지 않았다면, 스킬이 옳은 것을 가르치는지 알 수 없다."
— skills/writing-skills/SKILL.md:16
이 문장은 TDD의 핵심 원칙인 "테스트가 실패하는 것을 보지 않았다면 테스트가 옳은 것을 테스트하는지 알 수 없다"를 문서에 그대로 적용한 것이다.

이 두 층위가 합쳐질 때 TDD는 Superpowers의 척추가 된다: 코드를 만드는 과정과, 코드를 만드는 방법을 가르치는 문서를 만드는 과정이 동일한 규율로 묶인다. 규율의 전파가 자기 참조적(self-referential)이 된다.

아키텍처 다이어그램

아래 다이어그램은 두 개의 병렬 레인으로 구성된다. 왼쪽은 코드 TDD 사이클, 오른쪽은 스킬 문서에 매핑된 TDD 사이클이다. 두 레인의 구조는 동일하다 — RED, GREEN, REFACTOR 세 단계가 같은 논리로 반복된다.

두 레인이 동일한 색상 구조를 공유하는 것은 의도적이다. RED(붉은색)는 실패 관찰, GREEN(초록색)은 최소 충족, BLUE(파란색)는 정제. 스킬 레인의 "프로덕션 코드"는 SKILL.md이고, "테스트"는 서브에이전트 압박 시나리오다. 두 레인 모두 철칙(IRON) 노드에서 파생된다: 실패를 먼저 보지 않으면 다음 단계로 진입하지 않는다.

핵심 메커니즘 해설

1. 진짜 RED-GREEN-REFACTOR — 순서가 규율이다

TDD의 가장 흔한 위반은 "테스트를 나중에 쓰면 같은 결과"라는 합리화다. test-driven-development/SKILL.md는 이를 직접 반박한다:

  • RED 단계: 테스트를 먼저 쓰는 것은 인터페이스를 먼저 설계하는 것이다. "wished-for API"를 먼저 표현하면 구현 전에 사용 방식을 확정할 수 있다.
  • 실패 확인 (MANDATORY): 테스트가 통과하면 기존 동작을 테스트하는 것이다. "테스트가 실패하지 않으면 테스트가 없는 것"이라는 논리가 성립한다.
  • GREEN 단계: 테스트를 통과하는 최소한의 코드만 쓴다. YAGNI를 강제하는 구조적 장치다. 옵션 파라미터, 백오프 전략 — 테스트가 요구하기 전에는 없다.
  • REFACTOR 단계: 그린 상태에서만 리팩토링한다. 테스트가 레드인 상태에서 코드를 정리하면 "어느 변경이 테스트를 통과시켰는지" 모른다.

핵심은 순서다. 사이클을 건너뛰는 모든 이유 — "너무 간단해", "수동 테스트 했어", "X시간 지웠기엔 아까워" — 는 스킬 파일이 "Common Rationalizations" 표에 명시적으로 포착하여 반박한다.

2. 테스팅 안티패턴 — 모킹이 테스트를 죽이는 방식

TDD를 따르더라도 테스트가 "진짜 동작"을 검증하지 않으면 의미가 없다. testing-anti-patterns.md는 세 가지 철칙으로 이를 보호한다:

  • 안티패턴 1 — 모크 동작 테스트: screen.getByTestId('sidebar-mock')를 단언하는 테스트는 모크가 존재하는지를 테스트하는 것이다. 컴포넌트의 실제 동작을 테스트하지 않는다. TDD를 따랐다면 이 실수가 드러난다: 실제 코드 없이 테스트를 실행하면 모크 단언은 통과한다 — RED를 볼 수 없다는 뜻이다.
  • 안티패턴 2 — 프로덕션 클래스에 테스트 전용 메서드: destroy()가 테스트에서만 사용된다면 프로덕션 클래스가 아닌 테스트 유틸리티에 있어야 한다. 객체 생명주기와 테스트 정리는 다른 관심사다.
  • 안티패턴 3 — 이해 없이 모킹: 모킹할 메서드의 부수 효과를 이해하지 못하면 테스트가 의존하는 부수 효과도 함께 제거된다. "안전하게 모킹하자"는 충동이 테스트를 망가뜨린다.

세 안티패턴 모두 "TDD를 제대로 따랐다면 발생하지 않는다"는 결론으로 수렴한다. RED를 먼저 보면 모크 단언이 RED를 만들지 않는다는 사실이 즉시 드러난다.

3. 메타 TDD — 스킬 문서 작성에 적용

Superpowers의 가장 독특한 지점은 여기다. writing-skills/SKILL.md는 스킬 문서를 작성하는 것 자체가 TDD라고 선언한다. 핵심 흐름은 다음과 같다:

  1. RED (기준선 실행): 스킬 없이 서브에이전트에게 압박 시나리오를 준다. 에이전트가 어떤 합리화를 사용해서 규칙을 위반하는지 정확한 발언을 기록한다. "테스트가 실패하는 것을 본다"에 해당한다.
  2. GREEN (최소 스킬 작성): 기록한 위반 패턴만 다루는 스킬을 작성한다. 가상의 케이스를 위한 내용을 추가하지 않는다. 같은 시나리오를 스킬이 있는 상태에서 다시 실행해 에이전트가 준수하는지 확인한다.
  3. REFACTOR (허점 닫기): 에이전트가 새로운 합리화를 찾으면 그것을 명시적으로 스킬에 추가한다. 합리화 표(Rationalization Table)와 Red Flags 목록이 이 단계의 산출물이다.

이 메타 TDD가 없으면 스킬 문서는 "작성자에게 명확한 것"이 된다. 작성자가 당연하게 여기는 것을 독자(에이전트)가 건너뛰거나 합리화할 수 있다는 사실을 모른다. 서브에이전트 압박 테스트만이 이 간극을 드러낸다.

실제 발췌

"직접 만든다면" — 행동 형성 문서도 테스트해야 한다

TDD를 코드에만 적용하다가 Superpowers의 메타 원칙을 놓치는 경우가 많다. 핵심 교훈은 이것이다: 검증되지 않은 프롬프트는 검증되지 않은 코드다.

기준선(baseline) 없이 작성한 스킬은 의미가 없다

"이 스킬 문서는 명확하다"는 느낌은 작성자의 착각이다. 작성자는 이미 규칙을 알고 있기 때문에 문서가 명확하게 느껴진다. 에이전트는 규칙을 모르는 상태에서 문서를 읽고, 압박을 받으면 허점을 찾는다.

기준선 실행이 없다면 두 가지를 모른다:

  • 에이전트가 어떤 합리화를 쓰는지: 문서가 그 합리화를 막는지 알 수 없다.
  • 스킬이 실제로 동작 변화를 만드는지: 측정값이 없으니 개선도 없다.

Superpowers의 TDD Mapping에서 "Watch it fail"이 "Document exact rationalizations agent uses"로 매핑되는 이유가 여기 있다. 막연히 "실패했다"가 아니라, 실패할 때 에이전트가 쓴 정확한 표현을 기록해야 한다. 그 표현이 스킬의 Rationalization Table이 된다.

최소 구현 원칙 — 스킬에도 YAGNI

GREEN 단계에서 "최소 코드"라는 원칙은 스킬에도 그대로 적용된다: 기준선 테스트에서 발견된 위반 패턴만 다룬다. 가상의 미래 케이스를 예측해서 스킬을 늘리지 않는다. REFACTOR 단계에서 새로운 합리화가 발견되면 그때 추가한다.

이 원칙을 어기면 두 가지 문제가 생긴다: 스킬 파일이 비대해져 토큰 비용이 올라가고, 실제로 검증되지 않은 내용이 섞여 들어간다. Superpowers가 token efficiency를 별도 섹션(SDO)으로 강조하는 이유다.

자신의 프롬프트를 압박 테스트하라

직접 에이전트 워크플로를 구성하거나, 팀용 행동 지침을 작성한다면 동일한 원칙이 적용된다:

  1. 기준선 실행: 지침 없이 서브에이전트에게 전형적인 태스크를 준다. 어디서 어긋나는지, 어떤 표현으로 규칙을 우회하는지 기록한다.
  2. 최소 지침 작성: 발견된 위반만 다루는 지침을 작성한다. 같은 시나리오를 지침 있는 상태로 재실행한다.
  3. 반복: 새로운 우회 표현이 나오면 지침에 추가한다. "5회 이상 반복에서 동일한 형태로 수렴"할 때까지 계속한다.
핵심 교훈: 행동 형성 문서(스킬, 지침, 프롬프트)를 작성할 때 "명확하게 썼다"는 느낌은 테스트를 대체하지 않는다. 코드에서 "수동 테스트 했다"가 자동화 테스트를 대체하지 못하는 것과 정확히 같은 이유다. 에이전트가 스킬 없이 실패하는 것을 보지 않았다면, 스킬이 옳은 것을 가르치는지 알 수 없다.
