Module 0

전체 조감도

Superpowers의 4레이어 구조 — "부품이 어디에 있는지" 지도부터 읽어야 깊이 파고들 수 있다

왜 이 개념인가

Superpowers를 처음 접하면 "CLAUDE.md에 마크다운 파일 몇 개를 추가하는 것 아닌가?"라는 인상을 받기 쉽다. 실제로는 그렇지 않다. Superpowers는 에이전트 런타임(Claude Code, Codex, Gemini CLI…) 위에 올라가는 독립된 방법론 레이어다. 런타임이 "어떻게 API를 호출하고 툴을 실행하는가"를 담당한다면, Superpowers는 "에이전트가 어떤 절차로 생각하고 행동해야 하는가"를 강제한다.

이 분리가 왜 중요한가? 런타임이 바뀌어도 방법론은 유지된다. 실제로 plugin.json에 선언된 하나의 플러그인이 Claude Code, Codex App, Codex CLI, Cursor, Gemini CLI, GitHub Copilot CLI, Kimi Code, Factory Droid, Antigravity, OpenCode, Pi — 11개 하니스에서 동일하게 동작한다. 런타임마다 훅과 플러그인 포맷이 다르지만, skills/ 디렉터리 안의 마크다운 스킬들은 모든 하니스에서 그대로 읽힌다.

사용자 관점에서는 단순히 "설치만 하면 알아서 발동"으로 보이지만, 빌더 관점에서는 네 개의 명확한 레이어가 작동하고 있다. 이 조감도 모듈에서 먼저 지도를 그리고, 이후 모듈에서 각 부품을 확대해 살펴볼 것이다.

핵심 통찰: Superpowers는 "툴 설치"가 아니라 에이전트의 의사결정 구조를 덮어쓰는 방법론 주입이다. 스킬 파일들은 LLM이 실행할 절차적 지침이며, 훅은 그 지침이 매 세션마다 반드시 에이전트 컨텍스트에 존재하도록 보장하는 장치다.

아키텍처 다이어그램

아래 다이어그램은 Superpowers의 4레이어 구조와 제어 흐름을 보여준다. 각 박스 안의 이탤릭은 실제 파일·디렉터리 경로다. Layer 1(부트스트랩 훅)이 세션 시작 시 Layer 2(라우터 스킬)를 에이전트 컨텍스트에 주입하고, Layer 2는 Layer 3(스킬 라이브러리)로 요청을 라우팅하며, 스킬들이 순서대로 연결되어 Layer 4(방법론 파이프라인)를 형성한다.

핵심 메커니즘 해설

Layer 1 — 부트스트랩 훅 (hooks/)

훅은 에이전트 런타임이 특정 이벤트(세션 시작, 컴팩션 등)에 실행하는 외부 스크립트다. hooks/hooks.json에는 SessionStart 이벤트에 반응하는 훅이 등록되어 있고, matcher 패턴 startup|clear|compact에 해당하는 상황에서 실행된다. 실제 스크립트인 hooks/session-start는 skills/using-superpowers/SKILL.md를 읽어 그 내용을 JSON으로 직렬화한 뒤 에이전트의 additionalContext로 주입한다. 하니스마다 JSON 필드명이 다르기 때문에 스크립트는 환경변수를 보고 hookSpecificOutput.additionalContext(Claude Code), additional_context(Cursor), additionalContext(기타)를 구분해 출력한다.

Layer 2 — 라우터 스킬 (skills/using-superpowers/SKILL.md)

훅이 주입하는 유일한 스킬이 바로 using-superpowers다. 이 스킬은 두 가지 핵심 역할을 한다. 첫째, 스킬 발동 규칙(1% 규칙): "관련될 가능성이 1%라도 있으면 반드시 해당 스킬을 발동하라." 에이전트가 스스로 판단을 합리화하여 스킬을 건너뛰는 것을 막는 Red Flags 표와 함께 제시된다. 둘째, 플랫폼별 스킬 발동 방법: Claude Code에서는 Skill 툴, Codex에서는 네이티브 로딩, Gemini CLI에서는 activate_skill 툴을 쓴다. 이 라우터 스킬이 없으면 나머지 13개 스킬은 디스크에만 존재하고 에이전트는 그것을 쓸 규칙을 모른다.

Layer 3 — 스킬 라이브러리 (skills/)

14개의 스킬이 방법론별로 구분된다. 각 스킬은 skills/<name>/SKILL.md 형태로 저장된다.

분류 스킬 한 줄 설명 (frontmatter description)
방법론 (프로세스) brainstorming 창의적 작업 전 반드시 사용 — 구현 전 의도·요구사항·설계를 대화로 탐색
writing-plans 스펙·요구사항이 있을 때 코드 전에 사용 — 멀티스텝 구현 계획 작성
subagent-driven-development 현재 세션에서 독립 태스크를 포함한 구현 계획 실행 시 사용
executing-plans 별도 세션에서 리뷰 체크포인트와 함께 기존 계획 실행 시 사용
dispatching-parallel-agents 공유 상태나 순차 의존성 없이 병렬 처리 가능한 태스크 2개 이상일 때 사용
품질 (검증) test-driven-development 기능·버그픽스 구현 전 구현 코드 작성 전에 사용
systematic-debugging 버그·테스트 실패·예기치 않은 동작 발생 시 수정 제안 전에 사용
verification-before-completion 작업이 완료·수정·통과되었다고 주장하기 전, 커밋·PR 전에 사용
requesting-code-review 태스크 완료·주요 기능 구현·머지 전 작업이 요구사항을 충족하는지 검증 시 사용
협업 receiving-code-review 코드 리뷰 피드백 수신 시, 제안 구현 전 사용 — 기술적 엄밀성·검증 필요
using-git-worktrees 현재 워크스페이스와 격리된 피처 작업 시작 또는 구현 계획 실행 전 사용
메타 finishing-a-development-branch 구현 완료·테스트 통과 후 작업 통합 방법 결정 시 사용
writing-skills 새 스킬 생성·기존 스킬 편집·배포 전 스킬 동작 검증 시 사용
using-superpowers 모든 대화 시작 시 사용 — 스킬 발견·사용 방법 확립, 라우터 역할

Layer 4 — 방법론 파이프라인

스킬들이 순서대로 자동 연쇄하여 하나의 개발 파이프라인을 형성한다. README.md에 기술된 "The Basic Workflow"가 바로 이 파이프라인이다: brainstorming(아이디어 → 스펙) → using-git-worktrees(격리 워크스페이스) → writing-plans(구현 계획) → subagent-driven-development 또는 executing-plans(실행) → test-driven-development(RED-GREEN-REFACTOR) → requesting-code-review(리뷰) → finishing-a-development-branch(머지·PR·정리). 각 단계 사이에 verification-before-completion이 완료 주장 전에 개입한다.

실제 발췌

"직접 만든다면" 교훈

이 구조를 직접 구현한다면 최소 세 가지 요소가 필요하다. Superpowers의 설계를 분석해 보면 이 세 요소가 어떻게 협력하는지 명확하게 보인다.

요소 1: 부트스트랩 (훅)

에이전트가 세션을 시작할 때마다 방법론 지침이 컨텍스트에 존재해야 한다. 훅이 없으면 사용자가 매번 수동으로 지침을 붙여 넣어야 한다. 중요한 설계 결정은 세션 시작뿐 아니라 컴팩션 시에도 재주입하는 것이다 (matcher: "startup|clear|compact"). 컨텍스트 윈도우가 초기화되더라도 방법론은 다시 주입된다.

요소 2: 라우터 스킬 (메타 스킬)

스킬 라이브러리가 아무리 풍부해도 "언제 어떤 스킬을 쓰는가" 규칙이 없으면 에이전트는 스킬을 건너뛰는 합리화를 찾아낸다. Superpowers는 이를 using-superpowers 라는 전용 메타 스킬로 해결한다. 모든 스킬보다 먼저 로드되어 "1% 가능성만 있어도 무조건 스킬 발동"이라는 규칙과 Red Flags 표를 제공한다.

요소 3: 스킬 라이브러리 (마크다운 프로시저)

각 스킬은 단순한 문서가 아니라 에이전트가 따라야 할 절차적 지침이다. frontmatter의 description은 언제 발동할지를 알려주는 트리거 조건이고, 본문은 에이전트가 실행할 단계별 체크리스트다. 스킬이 마크다운 파일인 덕분에 런타임에 의존하지 않고 모든 하니스에서 동일하게 동작한다.

의사코드로 본 최소 방법론 플러그인 골격

my-methodology-plugin/
├── hooks/
│   └── session-start          # (1) 부트스트랩 — 세션마다 라우터 스킬 주입
├── skills/
│   ├── router/SKILL.md        # (2) 라우터 — "언제 어떤 스킬을?"
│   ├── workflow-a/SKILL.md    # (3) 스킬 라이브러리 — 실제 방법론 절차
│   └── workflow-b/SKILL.md
└── .harness-plugin/
    └── plugin.json            #     메타데이터 (name, version, hooks 등록)

이 골격에서 부트스트랩이 없으면 라우터가 작동하지 않고, 라우터가 없으면 스킬 라이브러리는 디스크에만 존재하는 파일이 된다. 세 요소가 모두 있어야 "설치만 하면 알아서 발동"이라는 사용자 경험이 완성된다.

빌더를 위한 핵심: Superpowers의 복잡성은 스킬 내용에 있지, 구조에 있지 않다. 구조 자체는 단순하다 — 훅 + 라우터 + 스킬 라이브러리. 중요한 것은 각 스킬이 에이전트 행동을 얼마나 정확하게 형성(shape)하는가이다. 이것이 이후 모듈들에서 깊이 파고들 주제다.
