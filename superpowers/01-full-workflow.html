Module 1

"Let's build X"의 전체 흐름

사용자가 "react todo 만들자"라고 입력한 순간부터 코드가 머지될 때까지 — 스킬들이 어떻게 자동 연쇄되는가

왜 이 개념인가

Module 0에서 우리는 Superpowers의 4레이어 구조를 지도로 그렸다. 이제 그 지도 위에 실제 개발 흐름 하나가 어떻게 진행되는지를 추적해 보자.

Superpowers를 처음 접하면 "스킬 파일들이 있는 것"처럼 보인다. 하지만 진짜 차별점은 개별 스킬이 아니라 스킬들이 자동으로 연쇄되는 방식에 있다. 각 스킬은 자신이 끝날 때 반드시 다음 스킬을 명시적으로 호출하도록 설계되어 있다. 에이전트가 임의로 단계를 건너뛰거나 순서를 바꾸지 못한다.

Superpowers의 공식 수락 테스트(acceptance test)는 이것이다:

"Let's make a react todo list"

이 한 문장을 새 세션에 입력했을 때, 에이전트가 코드를 바로 작성하지 않고 brainstorming 스킬을 자동 발동해 먼저 무엇을 만들지 탐색하기 시작하면 — 그것이 Superpowers가 제대로 동작한다는 증거다. 이 모듈에서는 그 순간부터 완료까지의 전체 상태 전이를 해부한다.

이 학습자료 자체가 산 증거다: 지금 읽고 있는 이 사이트를 만들 때도 동일한 파이프라인이 사용됐다. brainstorming으로 콘텐츠 구조를 협의하고, writing-plans로 모듈별 태스크 계획을 작성하고, subagent-driven-development로 각 HTML 파일을 구현하는 흐름 — 이것이 "추상적인 설명"이 아니라 이 문서 자체를 생산한 실제 방법론이다.

핵심 메커니즘 해설

Terminal State 규칙 — 각 스킬의 유일한 출구

Superpowers 파이프라인이 "에이전트가 임의로 단계를 건너뛰지 못하는" 이유는 각 스킬이 자신의 terminal state(종착 상태)를 명시적으로 제한하기 때문이다. 이것이 이 파이프라인을 "상태 기계(state machine)"로 볼 수 있는 근거다.

스킬 Terminal State (유일한 출구) 금지된 전이
brainstorming writing-plans 호출만 허용 frontend-design, mcp-builder 등 구현 스킬 직접 호출 금지
writing-plans subagent-driven-development 또는 executing-plans (사용자 선택) 직접 코드 작성 금지
subagent-driven-development finishing-a-development-branch (최종 리뷰 후) 리뷰 없이 다음 태스크 진행 금지
executing-plans finishing-a-development-branch 블로커 상태에서 추측하여 계속 진행 금지
verification-before-completion 검증 명령 실행 후 결과 확인 "아마 통과할 것"과 같은 가정 기반 완료 주장 금지

HARD-GATE — brainstorming의 불통과 장벽

파이프라인 전체에서 가장 강력한 게이트는 brainstorming 스킬 첫 번째 항목이다. 이 게이트는 설계 승인 없이는 어떤 구현 행동도 허용하지 않는다 — 프로젝트 크기와 무관하게, "단순하다"는 판단과 무관하게:

"Do NOT invoke any implementation skill, write any code, scaffold any project, or take any implementation action until you have presented a design and the user has approved it. This applies to EVERY project regardless of perceived simplicity."
— skills/brainstorming/SKILL.md:13-14

이 HARD-GATE가 존재하는 이유를 스킬은 직접 설명한다: "Simple projects are where unexamined assumptions cause the most wasted work." 간단해 보이는 todo 앱조차 "어떤 상태 관리 라이브러리?", "로컬 스토리지 vs 서버?", "타입스크립트 사용 여부?" 같은 결정이 필요하다. 이 결정들이 설계 단계에서 명시되지 않으면 구현 후 리워크로 돌아온다.

writing-plans의 Execution Handoff

writing-plans 스킬이 플랜 문서를 저장한 뒤 실행 단계로 넘어가는 방식은 단순히 "다음 스킬을 쓰세요"가 아니다. 사용자에게 두 가지 실행 방식을 명확히 제시하고 선택권을 준다. 이 선택이 파이프라인의 분기점이 된다:

"Plan complete and saved to docs/superpowers/plans/<filename>.md. Two execution options:

1. Subagent-Driven (recommended) — I dispatch a fresh subagent per task, review between tasks, fast iteration

2. Inline Execution — Execute tasks in this session using executing-plans, batch execution with checkpoints

Which approach?"
— skills/writing-plans/SKILL.md:160-166

서브에이전트 방식이 권장되는 이유: 각 태스크마다 신선한 컨텍스트를 가진 서브에이전트가 디스패치되므로 이전 태스크의 편향이나 오류가 누적되지 않는다. 또한 태스크 완료 후 즉시 리뷰(spec 준수 + 코드 품질)가 따르므로 조기에 문제를 포착한다.

subagent-driven-development의 태스크 루프

실행 단계에서 서브에이전트 방식을 선택했을 때, 각 태스크는 다음 순환을 거친다:

  1. 구현 서브에이전트 디스패치 — 플랜에서 해당 태스크만 추출해 신선한 컨텍스트로 시작
  2. 구현 완료 보고 — DONE / DONE_WITH_CONCERNS / NEEDS_CONTEXT / BLOCKED 중 하나
  3. 리뷰 서브에이전트 디스패치 — diff 파일을 전달, spec 준수 + 코드 품질 두 가지 평가
  4. 이슈 발견 시 — Critical/Important 수준이면 수정 서브에이전트 재디스패치 후 재리뷰
  5. 승인 시 — 태스크 완료 기록, 다음 태스크로

이 루프가 모든 태스크를 통과하면 최종 whole-branch 코드 리뷰가 한 번 더 실행되고, 그 후 finishing-a-development-branch 스킬로 전이된다.

verification-before-completion — 파이프라인의 공통 경비

이 스킬은 파이프라인의 특정 단계가 아니라 모든 완료 주장 직전에 개입한다. "증거 없는 주장은 거짓이다"라는 원칙을 코드화한 것이다. 각 서브에이전트의 태스크 완료 보고, 전체 파이프라인의 완료 선언 — 어느 것도 실제 검증 명령 실행 없이는 성립하지 않는다.

이 학습자료가 통과한 파이프라인

지금 읽고 있는 이 사이트 자체가 동일한 파이프라인의 산물이다:

  1. brainstorming — "Superpowers 학습 사이트를 만들자"에서 시작해 모듈 구성, 발췌 방식, 한국어 교육 스타일 등을 협의해 스펙 문서 작성
  2. writing-plans — 9개 모듈 각각의 HTML 파일, 공통 자산(CSS, style.css), 태스크별 brief/report 구조를 단계별 계획으로 분해
  3. subagent-driven-development — 각 모듈 HTML을 독립 태스크로 디스패치, 인용 검증 포함 self-review 후 커밋

"이론"이 아니라 이 문서가 존재한다는 사실 자체가 파이프라인의 동작 증거다.

실제 발췌

"직접 만든다면" 교훈

Superpowers 파이프라인의 핵심 설계 원칙을 추출하면, 이것은 각 스킬이 다음 스킬을 명시적으로 호출하는 상태 기계다. 일반적인 체크리스트나 가이드라인과 다른 점은 네 가지다.

원칙 1: 출구를 명시적으로 제한하라

brainstorming 스킬이 "다음으로 writing-plans를 쓰세요"라고만 했다면 에이전트는 "이번엔 간단하니까 직접 구현해도 되겠다"라고 합리화할 수 있다. Superpowers는 이를 두 겹으로 막는다:

  • HARD-GATE: 구현 행동 자체를 명시적으로 금지
  • Terminal state 선언: "The ONLY skill you invoke is writing-plans"

출구가 하나뿐인 상태 기계는 에이전트의 합리화 경로를 원천 차단한다.

원칙 2: 사용자 게이트를 명시적 체크포인트로 만들어라

파이프라인에는 두 개의 주요 사용자 게이트가 있다:

  1. 설계 승인 (brainstorming → writing-plans 전환 전)
  2. 실행 방식 선택 (writing-plans → 실행 스킬 전환 시)

이 게이트들이 "묵시적 동의"(응답 없으면 계속 진행)가 아니라 명시적 응답을 요구하는 이유: 에이전트가 잘못된 방향으로 한 시간을 달리다가 결국 처음부터 다시 하는 비용을 방지하기 위해서다.

원칙 3: 서브에이전트는 컨텍스트 오염을 방지한다

태스크 N의 구현 실수나 선택이 태스크 N+1의 에이전트 판단에 영향을 주지 않도록, 각 태스크는 신선한 컨텍스트를 가진 서브에이전트로 실행된다. 컨트롤러(오케스트레이터)는 "각 서브에이전트가 필요한 것만" 정밀하게 전달한다: 

컨트롤러가 구현 서브에이전트에게 전달하는 것:
  ① 이 태스크가 프로젝트에서 어떤 위치인지 (1줄)
  ② 태스크 brief 파일 경로 (요구사항의 유일한 출처)
  ③ 이전 태스크들이 결정한 인터페이스 (brief가 모를 수 있는 것)
  ④ 리포트 파일 경로 (서브에이전트가 결과를 기록할 곳)

컨트롤러가 전달하지 않는 것:
  ✗ 이전 태스크들의 전체 대화 기록
  ✗ 수정된 의사결정 이력
  ✗ 태스크 1-N의 누적 컨텍스트 요약

원칙 4: 진행 상태를 파일에 기록하라

LLM의 컨텍스트 메모리는 세션 압축(compaction) 시 소실된다. subagent-driven-development 스킬은 이를 해결하기 위해 .superpowers/sdd/progress.md 파일에 완료된 태스크를 기록하도록 강제한다. 컨텍스트 압축 후 재개 시 에이전트는 이 파일을 먼저 확인하고 이미 완료된 태스크를 다시 실행하는 실수를 방지한다.

상태 기계로 보는 핵심 인사이트: Superpowers 파이프라인의 각 스킬은 입력(트리거 조건) → 처리(체크리스트) → 출력(terminal state)를 명시적으로 정의한다. 이 세 요소가 모두 명시된 스킬은 에이전트가 임의로 해석하거나 단계를 생략할 여지가 없다. 자신만의 방법론 플러그인을 만든다면 — 각 워크플로우 단계를 "다음 단계가 무엇인가"까지 포함해 명시해야 한다. "좋은 방법론"이 아니라 "강제되는 방법론"이어야 에이전트가 따른다.
