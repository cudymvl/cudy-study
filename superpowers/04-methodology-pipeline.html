Module 4

방법론 파이프라인

아이디어가 코드가 되기 전에 반드시 통과해야 하는 세 관문 — brainstorming → writing-plans → executing-plans — 과 각 게이트가 코드 품질을 높이는 구체적 이유

왜 이 개념인가

Superpowers를 쓰면 코드 퀄리티가 오른다는 말을 들을 때, 가장 먼저 드는 의문은 이것이다: "왜? 스킬 파일 몇 개가 어떻게 결과물의 품질을 바꾸는가?"

답은 스킬의 내용이 아니라 스킬이 강제하는 순서에 있다. Superpowers의 방법론 파이프라인은 세 가지 원칙을 코드를 쓰기 전에 강제한다:

  1. 코드 전에 설계 — brainstorming 스킬의 HARD-GATE가 설계 승인 없이는 구현을 시작할 수 없게 막는다.
  2. 작고 검증 가능한 단위 — writing-plans의 bite-sized 태스크 원칙이 2~5분짜리 단계로 작업을 쪼갠다. 각 단계는 독립적으로 테스트되고 커밋된다.
  3. 빈번한 커밋 — 각 태스크 끝에 커밋을 명시적으로 요구하며, 이는 되돌아갈 수 있는 체크포인트를 촘촘하게 만든다.

이 세 원칙은 에이전트가 "빠르게 일단 코드를 쏟아내는" 패턴을 차단한다. 대부분의 AI 코드 품질 문제는 잘못된 가정을 검증 없이 구현한 데서 온다. 파이프라인은 그 가정들을 설계 단계에서 노출시키고, 작은 단위로 쪼개어 각 단계마다 검증을 강제함으로써 오류가 누적되기 전에 발견될 수 있게 한다.

게이트(주황색 마름모)는 단순한 체크포인트가 아니다. brainstorming의 HARD GATE는 스킬 파일에 명시적으로 선언되어 있어 에이전트가 "이 정도는 너무 간단해서 설계가 필요 없다"는 합리화를 차단한다. writing-plans의 게이트는 플랜 완성도(No Placeholders, Self-Review)를 통과해야 열린다.

핵심 메커니즘 해설

1. brainstorming — "너무 간단해" 안티패턴 차단

brainstorming 스킬의 가장 중요한 규칙은 HARD-GATE다. 코드·스캐폴딩·구현 액션을 "설계 제시 + 유저 승인" 전에 절대 시작하지 말 것을 강제한다. 이것은 "모든 프로젝트에, 지각된 단순함에 관계없이(EVERY project regardless of perceived simplicity)" 적용된다.

Anti-Pattern 섹션은 한 걸음 더 나간다: 간단한 프로젝트일수록 검증 없는 가정이 낭비를 만든다고 명시한다. 설계 자체가 짧아도 되지만(a few sentences for truly simple projects), 반드시 제시하고 승인을 받아야 한다.

세 가지 핵심 원칙이 brainstorming을 품질 게이트로 만든다:

  • 한 번에 한 질문 (One at a time): 복수의 질문을 동시에 던지지 않는다. 한 토픽이 더 탐구가 필요하면 여러 메시지로 나눈다. 이 규칙은 유저가 압도되는 것을 막고, 에이전트가 답변을 제대로 이해했는지 확인할 수 있게 한다.
  • YAGNI 무자비하게 적용: 모든 설계에서 불필요한 기능을 제거한다. 에이전트의 자연스러운 경향 — "이왕 하는 김에 이것도" — 을 명시적으로 차단한다.
  • 2-3 접근법 항상 제안: 단일 해결책을 직접 제시하지 않는다. 트레이드오프와 추천 이유를 포함한 선택지를 먼저 제시해 유저가 실제 요구사항과 가장 맞는 방향을 고를 수 있게 한다.

2. writing-plans — zero-context 엔지니어 가정

writing-plans는 플랜을 읽을 엔지니어(혹은 다음 세션의 에이전트)가 "우리 코드베이스에 대해 거의 아무것도 모른다"고 가정하고 플랜을 작성하도록 요구한다. 이 가정이 플랜의 품질을 결정한다:

  • No Placeholders: "TBD", "TODO", "implement later", "add appropriate error handling" 같은 표현은 모두 플랜 실패(plan failures)로 분류된다. 모든 단계는 엔지니어가 실제로 필요한 내용을 담아야 한다 — 실제 코드, 실제 명령어, 예상 출력.
  • Bite-Sized (2-5분) 단위: "failing test 작성", "실패 확인을 위해 실행", "최소 구현", "통과 확인을 위해 실행", "커밋" — 이 다섯 단계가 하나의 TDD 사이클을 이룬다. 각 단계가 독립적이고 검증 가능한 액션이다.
  • TDD 스텝 명시: 플랜에 테스트 코드를 직접 포함시킨다. "위 내용에 대한 테스트를 작성하세요"라는 표현은 금지다. 실제 테스트 함수 코드가 플랜 안에 있어야 한다.

3. 유저 승인 게이트 — 다음 단계 진입 조건

파이프라인에는 두 개의 명시적 유저 승인 게이트가 있다:

게이트 위치 조건 실패 시
설계 게이트 brainstorming → writing-plans 사이 spec 작성 + Spec Self-Review + 유저 명시적 승인 변경 요청 반영 후 재검토, 승인 전까지 플랜 작성 금지
플랜 게이트 writing-plans → executing-plans 사이 No Placeholders 통과 + Self-Review 완료 + 유저 실행 방식 선택 플랜 보완 후 재제시, 승인 전까지 코드 실행 금지

두 게이트 모두 "유저가 명시적으로 진행하기 전까지는 다음 단계로 넘어가지 않는다"는 규칙을 스킬 파일에 텍스트로 선언해 에이전트 행동을 제약한다. writing-plans는 플랜 완성 후 "Subagent-Driven(권장)" vs "Inline Execution" 중 선택을 명시적으로 묻고, 유저의 응답을 기다린다.

4. executing-plans — 블로커에서 멈추기

실행 단계에서 품질을 지키는 핵심 규칙은 하나다: 블로커(의존성 누락, 테스트 실패, 지시 불명확)를 만나면 즉시 멈추고 물어보는 것. "추측하지 말 것(don't guess)"이 명시적 규칙이다. 에이전트가 막힌 상태에서 앞으로 밀어붙이는 것이 기술 부채의 주요 원인이기 때문이다.

실제 발췌

"직접 만든다면" — 3단 관문이 품질의 핵심인 이유

Superpowers 방법론 파이프라인을 자신의 워크플로에 적용할 때 가장 중요한 교훈은 하나다: 게이트의 수가 품질의 수준을 결정한다. 게이트가 없으면 에이전트는 (그리고 인간도) 빠른 경로로 달려가 가정을 검증하지 않은 채 수백 줄의 코드를 만들어낸다.

설계 게이트 — 코드를 쓰기 전에 멈추기

설계 게이트의 핵심은 에이전트가 "이건 너무 간단해서 설계가 필요 없다"고 느끼는 바로 그 순간에 발동해야 한다는 것이다. 간단한 것일수록 가정이 암묵적이고, 암묵적 가정이 가장 많은 낭비를 만든다. 직접 구현할 때도 "무엇을 만드는지 한 문단으로 써보고 승인받기"를 게이트로 삼으면 설계 단계에서 80%의 방향 오류를 잡을 수 있다.

플랜 게이트 — No Placeholders 강제하기

플랜 게이트는 실행 전 플랜의 완성도를 점검하는 관문이다. 실용적인 기준은 하나다: 이 플랜을 처음 보는 사람도 아무것도 추측하지 않고 실행할 수 있는가? "적절한 에러 처리 추가", "위의 테스트 작성" 같은 표현이 하나라도 있으면 플랜은 미완성이다. 실행자(에이전트 또는 인간)가 구멍을 채우기 위해 원래 요구사항과 다른 결정을 내리는 순간 품질이 저하된다.

실행 게이트 — 블로커에서 멈추기

실행 중 게이트는 블로커를 만났을 때 앞으로 밀어붙이지 않는 것이다. 에이전트의 기본 경향은 막힌 상태에서 "그럴듯한 대안"을 선택하는 것이다. 이 대안이 원래 설계와 충돌하면 기술 부채가 쌓인다. executing-plans는 이를 명시적 규칙으로 선언한다: 블로커에서 즉시 멈추고, 추측하지 말고, 물어볼 것.

3게이트 적용 요약:
  1. 설계 게이트 — "무엇을" 명문화하고 승인받기. 아무리 간단해도 예외 없이.
  2. 플랜 게이트 — "어떻게"를 No Placeholders 수준으로 구체화하기. 추측 여지 없애기.
  3. 실행 게이트 — 블로커에서 멈추기. 추측으로 앞으로 가는 것은 진전이 아니다.
이 세 관문이 순서대로 작동할 때, 에이전트의 속도와 인간의 방향성이 결합되어 코드 품질이 올라간다.
