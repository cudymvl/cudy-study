Module 2

스킬 해부학

스킬이 Superpowers의 원자(atom)인 이유 — SKILL.md 구조, frontmatter 규칙, 트리거 흐름, 그리고 스킬 3종을 낱낱이 해부한다

왜 이 개념인가

Module 1에서 "Let's build X"가 brainstorming → writing-plans → subagent-driven-development로 자동 연쇄되는 전체 파이프라인을 추적했다. 그런데 그 파이프라인의 각 단계를 구성하는 원자 단위가 무엇인가? 바로 스킬(skill)이다.

스킬을 그냥 "마크다운 파일"로 보면 Superpowers가 왜 강력한지 이해하기 어렵다. 스킬은 에이전트가 올바른 순간에 자동으로 찾아 적용하는 재사용 가능한 기법·패턴·레퍼런스의 저장소다. 구조를 이해해야 직접 스킬을 만들 수 있고, 기존 스킬이 왜 그런 형태로 작성되었는지도 납득된다.

이 모듈에서 해부하는 것은 세 가지다:

  1. 파일 구조 — SKILL.md + 지원 파일의 레이아웃과 존재 이유
  2. Frontmatter 규칙 — name/description의 정확한 역할과 작성 규칙
  3. 스킬 3종 — Technique / Pattern / Reference가 각각 무엇이고 어떻게 다른가
직접 만들기의 전제: writing-skills/SKILL.md는 "스킬을 만드는 스킬"이다. 이 모듈을 읽고 나면 최소 SKILL.md 골격을 직접 작성할 수 있는 수준이 목표다.

아키텍처 다이어그램

① 스킬 파일 구조 — SKILL.md + 지원 파일

모든 스킬은 하나의 디렉터리 아래 SKILL.md를 필수로 갖는다. 지원 파일은 "무거운 레퍼런스(100줄+)" 또는 "재사용 가능한 도구"일 때만 추가한다. 아래는 writing-skills/가 실제로 가진 지원 파일 목록이다.

② Frontmatter 구조 — 두 필드의 역할 분리

SKILL.md 상단의 YAML frontmatter는 두 개의 필수 필드만 갖는다. name은 스킬의 식별자이고, description은 에이전트가 "지금 이 스킬을 로드해야 하는가"를 판단하는 트리거 신호다. 

graph LR
    FM["<--- YAML frontmatter --->"]
    N["name:\nskill-name-with-hyphens"]
    D["description:\nUse when ... (트리거 조건만)"]
    BODY["# 스킬 본문\n(Overview, When to Use,\nCore Pattern, Quick Reference ...)"]

    FM --> N
    FM --> D
    FM -->|"같은 SKILL.md 안"| BODY

    D -->|"에이전트가 이것만 읽고\n로드 여부 결정"| BODY

③ 트리거 흐름 — description이 어떻게 매칭 신호가 되는가

에이전트는 전체 스킬 본문을 모두 읽지 않는다. 먼저 각 스킬의 description을 스캔해 "지금 이 스킬이 필요한가?"를 판단하고, 매칭된 스킬만 전체 로드한다. 그래서 description은 "스킬이 무엇을 하는가"가 아니라 "언제 이 스킬이 필요한가"를 기술해야 한다. 

flowchart TD
    TASK(["에이전트: 새 태스크 수신"])
    SCAN["모든 스킬의 description 스캔"]
    MATCH{"description이\n현재 상황과 매칭?"}
    LOAD["SKILL.md 전체 로드\n(Overview, Pattern, Reference 순)"]
    SKIP["스킬 건너뜀\n(토큰 절약)"]
    APPLY["기법 적용"]

    TASK --> SCAN
    SCAN --> MATCH
    MATCH -->|"Yes — 트리거 조건 일치"| LOAD
    MATCH -->|"No"| SKIP
    LOAD --> APPLY

핵심 메커니즘 해설

1. Frontmatter: name과 description의 정확한 규칙

name 필드에는 문자·숫자·하이픈만 사용한다. 괄호·특수문자는 금지다. 예: test-driven-development (O), TDD (approach) (X).

description 필드의 핵심 규칙은 두 가지다:

  • "Use when..."으로 시작 — 트리거 조건에 집중하게 하는 문장 형식
  • 워크플로 요약 금지 — 에이전트가 description만 읽고 스킬 본문을 건너뛰는 함정을 방지한다

이 두 번째 규칙이 중요한 이유를 writing-skills/SKILL.md는 실제 실험으로 설명한다: description에 "code review between tasks"라고 워크플로를 요약했더니 에이전트가 리뷰를 한 번만 실시했다. 스킬 본문에는 두 단계 리뷰(spec 준수 + 코드 품질)가 명시되어 있었는데도. description을 트리거 조건만으로 바꾸자 에이전트가 본문을 정확히 읽고 두 단계를 모두 수행했다.

"**The trap:** Descriptions that summarize workflow create a shortcut agents will take. The skill body becomes documentation agents skip."
— skills/writing-skills/SKILL.md:158

2. 스킬 3종 — Technique / Pattern / Reference

Superpowers는 스킬을 성격에 따라 세 가지로 분류한다. 이 분류는 테스트 방법까지 달라지므로 단순한 레이블이 아니다.

종류 정의 예시 테스트 방법
Technique 따라야 할 단계가 있는 구체적 방법 condition-based-waiting, root-cause-tracing 새 시나리오에 기법을 올바르게 적용하는가?
Pattern 문제를 바라보는 사고 방식 flatten-with-flags, test-invariants 패턴 적용 여부를 올바르게 인식하는가?
Reference API 문서·구문 가이드·도구 레퍼런스 office docs, library guides 필요한 정보를 찾아 정확히 사용하는가?

3. 디렉터리 구조 + Progressive Disclosure

모든 스킬은 flat namespace에 존재한다 — 중첩 폴더 없이 skills/ 아래에 스킬 이름 하나가 바로 온다. 이렇게 하면 에이전트가 전체 스킬 목록을 하나의 grep으로 탐색할 수 있다.

지원 파일은 "필요할 때만" 추가한다는 점이 Progressive Disclosure의 핵심이다. 에이전트는 SKILL.md만 먼저 읽고, 무거운 레퍼런스가 필요해질 때 비로소 anthropic-best-practices.md나 testing-skills-with-subagents.md 같은 지원 파일을 로드한다. 이렇게 하면 매 대화마다 수백 KB의 문서가 컨텍스트에 자동 삽입되는 것을 방지한다.

지원 파일을 별도로 분리하는 기준은 두 가지다:

  • Heavy reference — 100줄 이상의 API 문서·종합 구문 가이드
  • Reusable tools — 스크립트, 유틸리티, 템플릿 (코드로 직접 실행 가능한 것)

그 외 원칙·개념·50줄 이하 코드 패턴은 SKILL.md 인라인에 둔다.

4. 스킬 = "재사용 기법", 서사가 아니다

writing-skills/SKILL.md는 이 점을 명시적으로 못 박는다:

"Skills are: Reusable techniques, patterns, tools, reference guides
Skills are NOT: Narratives about how you solved a problem once"
— skills/writing-skills/SKILL.md:25-28

"2025-10-03 세션에서 우리가 발견한 것"처럼 특정 세션을 서술하는 스킬은 다음 에이전트가 재사용할 수 없는 일회성 서사다. 스킬은 패턴을 추출한 것이어야 한다 — 어떤 프로젝트·어떤 에이전트에서도 적용 가능한 형태로.

실제 발췌

"직접 만든다면" — 최소 SKILL.md 골격

위에서 해부한 규칙들을 종합하면, 새 스킬을 만들 때 작성해야 하는 최소 SKILL.md 골격은 다음과 같다. 각 섹션에 주석으로 "왜 이 섹션이 필요한가"를 달았다.

지원 파일을 추가해야 할 때

SKILL.md 인라인으로 충분한지 판단하는 기준:

  • 레퍼런스 섹션이 100줄을 넘는다 → 별도 reference.md로 분리하고 SKILL.md에서 링크
  • 에이전트가 직접 실행할 수 있는 스크립트/유틸리티가 있다 → 별도 파일로 분리
  • 코드 예제가 50줄 미만이고 개념 설명과 붙어있을 때 → 인라인 유지
핵심 인사이트: SKILL.md를 작성할 때 가장 많이 틀리는 부분은 description이다. "이 스킬은 X를 한다"고 설명하고 싶어지는 본능을 눌러야 한다. 에이전트는 이미 스킬이 무엇을 하는지 본문에서 읽을 것이다. description의 유일한 역할은 "지금 이 스킬을 꺼내야 하는가"를 0.1초 안에 판단하게 하는 것이다. 그 판단 기준은 상황과 증상이지, 스킬의 기능 요약이 아니다.
