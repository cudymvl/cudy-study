Module 3

부트스트랩 & 자동 트리거

스킬이 "알아서 발동"되는 이유 — SessionStart hook이 using-superpowers를 컨텍스트에 주입하고, 1% 규칙이 스킬 발동을 강제하는 전체 메커니즘

왜 이 개념인가

Module 2에서 스킬의 구조를 해부했다. 스킬은 잘 작성된 마크다운 파일이고, description이 트리거 신호 역할을 한다는 것도 알았다. 그런데 여기서 핵심 질문 하나가 남는다: 에이전트는 어떻게 스킬의 존재를 "처음부터" 알고 있는가?

이 질문에 대한 답이 없으면, 스킬은 디스크에 박제된 죽은 파일이다. Superpowers의 CLAUDE.md:76은 이것을 정확히 지적한다: "A real integration loads the using-superpowers bootstrap at session start. The bootstrap is what causes skills to auto-trigger at the right moments. Without it, the skills are dead weight — present on disk but never invoked."

이 모듈에서 설명하는 것은 그 부트스트랩 메커니즘의 전체 구조다:

  1. SessionStart hook — 세션 시작 시 자동으로 실행되는 bash 스크립트
  2. 컨텍스트 주입 — using-superpowers/SKILL.md를 읽어 JSON으로 에이전트에 전달
  3. 하니스 분기 — Cursor / Claude Code / Copilot CLI 등 플랫폼마다 다른 JSON 포맷
  4. 1% 규칙 — 주입된 using-superpowers가 에이전트를 "무조건 스킬 체크"로 강제
  5. 멀티하니스 이식 — references/*-tools.md로 도구 매핑을 추상화해 하니스 이식성 확보
수용 테스트(Acceptance Test): Superpowers의 CLAUDE.md는 새 하니스 통합의 성공 기준을 이것으로 정의한다 — 깨끗한 세션에서 "Let's make a react todo list"를 보내면 코드를 쓰기 전에 brainstorming 스킬이 자동으로 발동해야 한다. 발동하지 않으면 통합은 실패다.

핵심 메커니즘 해설

1. SessionStart hook이 하는 일

hooks/session-start는 세 가지 작업을 순서대로 수행한다:

  1. 파일 읽기: cat "${PLUGIN_ROOT}/skills/using-superpowers/SKILL.md"로 using-superpowers의 전체 내용을 읽는다. 읽기 실패 시 에러 메시지를 대신 반환해 set -euo pipefail에도 불구하고 세션이 중단되지 않도록 한다.
  2. JSON 이스케이프: escape_for_json() 함수가 역슬래시 → 따옴표 → 개행 → CR → 탭 순으로 5번의 bash parameter substitution을 수행한다. 주석이 명시하듯 이것은 문자 단위 루프보다 "orders of magnitude faster"하다.
  3. JSON 출력: printf로 플랫폼에 맞는 JSON을 생성하고 | cat으로 출력한다. heredoc 대신 printf를 사용하는 이유는 bash 5.3+의 heredoc hang 버그 때문이다 (이슈 #571).

출력된 JSON은 하니스가 LLM 컨텍스트에 자동으로 삽입한다. 결과적으로 에이전트는 대화 첫 메시지를 받기 전부터 이미 <EXTREMELY_IMPORTANT> 블록을 읽은 상태로 시작한다.

2. 하니스 감지 — 환경변수 세 가지

플랫폼 감지는 환경변수 존재 유무로만 판단한다. 순서가 중요하다:

우선순위 조건 플랫폼 출력 필드
1 CURSOR_PLUGIN_ROOT 존재 Cursor IDE additional_context (snake_case)
2 CLAUDE_PLUGIN_ROOT 존재 AND COPILOT_CLI 없음 Claude Code hookSpecificOutput.additionalContext (중첩)
3 그 외 (fallback) Copilot CLI (COPILOT_CLI=1) 또는 미지 플랫폼 additionalContext (SDK 표준, top-level)

Cursor가 CURSOR_PLUGIN_ROOT와 CLAUDE_PLUGIN_ROOT를 동시에 설정하는 경우가 있기 때문에 Cursor 감지를 먼저 확인한다. Claude Code도 CLAUDE_PLUGIN_ROOT를 설정하지만 COPILOT_CLI는 설정하지 않으므로 2번 조건이 Claude Code를 정확히 선별한다.

3. 1% 규칙 — 스킬 발동의 강제성

주입된 using-superpowers SKILL.md의 <EXTREMELY-IMPORTANT> 블록은 에이전트의 스킬 체크 기준을 "확실히 필요할 때"에서 "1%라도 관련될 수 있을 때"로 낮춘다. 이것이 자동 트리거의 두 번째 층이다.

SessionStart hook이 "스킬이 있다는 것"을 알려준다면, 1% 규칙은 "언제든 스킬을 꺼내야 한다는 것"을 강제한다. 두 층이 함께 작동해야 "Let's make a react todo list" 한 마디에 brainstorming이 코드보다 먼저 발동하는 결과가 만들어진다.

SKILL.md는 에이전트가 이 규칙을 회피하려 드는 구체적인 합리화 패턴을 "Red Flags" 표로 12가지 열거한다. 예를 들어 "This is just a simple question"이라는 생각이 드는 것 자체가 멈춰야 한다는 신호다 — 질문도 태스크이며, 스킬 체크가 선행되어야 한다.

4. 멀티하니스 이식 — references/*-tools.md

스킬 본문은 특정 하니스의 도구 이름을 직접 쓰지 않는다. "dispatch a subagent", "create a todo", "read a file"처럼 추상 동작으로 기술한다. 하니스별 도구 매핑은 references/ 아래 파일 6개가 담당한다:

  • claude-code-tools.md — Claude Code (Task tool, Bash, Read 등)
  • codex-tools.md — OpenAI Codex
  • copilot-tools.md — GitHub Copilot CLI
  • gemini-tools.md — Google Gemini CLI
  • pi-tools.md — Pi
  • antigravity-tools.md — Antigravity

using-superpowers SKILL.md가 이 파일들을 링크로 안내하면, 에이전트는 자신이 동작 중인 하니스에 맞는 파일만 로드한다. 하니스가 달라져도 스킬 본문을 고칠 필요가 없다. Gemini CLI는 더 나아가 GEMINI.md를 통해 도구 매핑을 세션 시작 시 자동 로드한다.

실제 발췌

"직접 만든다면" — 최소 SessionStart 주입 패턴

자신의 "라우터 문서"(how-to, 에이전트 지침 등)를 세션 시작 시 자동 주입하고 싶다면 session-start 패턴을 그대로 적용할 수 있다. 핵심 단계는 세 가지다:

주의점 — bash 5.3+ heredoc hang

위 코드에서 printf를 고집하는 이유가 있다. 소스 주석과 이슈 #571이 명시하듯, bash 5.3 이상에서는 heredoc을 파이프에 쓸 때 프로세스가 무한 대기(hang) 상태에 빠지는 버그가 있다. 다음 패턴은 쓰면 안 된다: 

# 위험: bash 5.3+에서 hang 가능
cat <<EOF | some_command
{"additionalContext": "$escaped"}
EOF

대신 반드시 printf '...' "$var" | cat 형태를 사용한다. | cat은 불필요해 보이지만, 일부 하니스 환경에서 출력 버퍼링 문제를 방지하기 위해 유지한다.

hooks.json 등록 — matcher 범위

hook 스크립트를 만들었으면 플랫폼의 hooks 설정 파일에 등록해야 한다. Claude Code라면 hooks.json의 matcher에 주의한다:

  • "startup|clear|compact" — 세 이벤트 모두 커버 (Superpowers 기본)
  • "startup"만 쓰면 /clear 후 컨텍스트가 사라졌을 때 재주입이 안 됨
  • async: false는 반드시 유지 — LLM이 주입 전에 응답을 시작하면 효과가 없음

멀티하니스를 지원하는 스킬을 쓴다면

직접 만드는 스킬 본문에서도 특정 하니스의 도구 이름을 직접 쓰지 말 것. "파일을 읽는다", "서브에이전트를 디스패치한다"처럼 추상 동작으로 기술하고, 하니스별 매핑 파일을 references/ 아래 분리하면 나중에 새 하니스를 지원할 때 스킬 본문은 건드리지 않아도 된다.

핵심 인사이트: SessionStart hook은 단순한 자동화 편의 기능이 아니다. 이것이 없으면 스킬은 에이전트가 직접 파일을 열어봐야만 존재를 알 수 있는 죽은 문서다. hook → using-superpowers 주입 → 1% 규칙의 세 층이 함께 작동할 때 비로소 "Let's build X" 한 마디에 brainstorming이 자동으로 발동하는 능동적인 워크플로 시스템이 만들어진다.
