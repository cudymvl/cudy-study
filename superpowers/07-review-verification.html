Module 7

리뷰 & 검증 루프

품질은 "완료를 주장하기 전 검증"에서 나온다 — 코드 리뷰 요청·수신의 규율, 증거 없는 완료 주장 금지, 브랜치 마무리 절차까지 네 스킬이 하나의 게이트 루프를 구성한다

왜 이 개념인가

AI 에이전트의 가장 고질적인 문제 중 하나는 환각성 완료 주장이다. 에이전트는 테스트를 실행하지 않고 "테스트가 통과합니다"라고 말하고, 빌드를 확인하지 않고 "빌드가 성공합니다"라고 말한다. 이 습관이 방치되면 인간 파트너는 "믿을 수 없어요"라는 신뢰 붕괴에 이른다. Superpowers는 이를 구조로 막는다.

네 스킬 — requesting-code-review, receiving-code-review, verification-before-completion, finishing-a-development-branch — 은 구현이 끝난 이후의 파이프라인을 다룬다. 구현 이후가 품질을 결정한다는 전제다: 아무리 잘 짠 코드도 리뷰·검증·정리 없이 머지되면 기술 부채와 신뢰 손상을 낳는다.

에이전트가 "다 됐어요"라고 거짓말하는 이유

에이전트는 확인 없이 완료를 주장하는 경향이 있다. 이는 악의가 아니라 구조의 부재다 — 명시적인 검증 게이트가 없으면 에이전트는 "코드를 변경했으니 고쳐졌겠지"라는 추론으로 완료를 선언한다. verification-before-completion은 이를 "증거 없는 주장 = 거짓말"로 규정하고, 검증 명령 실행 → 출력 확인이라는 의무적 절차를 부과한다.

코드 리뷰 측면도 마찬가지다. 리뷰어의 피드백을 감정적으로 수용하거나 맹목적으로 구현하면 잘못된 수정이 들어갈 수 있다. receiving-code-review는 "기술적 평가, 감정적 수행이 아님"을 원칙으로 삼아 각 피드백 항목을 먼저 코드베이스 현실에 맞게 검증하고 나서 구현하도록 강제한다.

게이트 없이는 루프가 없다

각 스킬은 독립적인 규율이 아니라 하나의 루프를 구성한다. 구현이 끝나면 코드 리뷰를 요청하고, 피드백을 기술적으로 수신·반영하고, 완료 전 검증으로 증거를 확보하고, 브랜치 마무리 절차로 정리한다. 어느 게이트에서도 실패하면 앞 단계로 돌아간다. 이 루프가 있어야 "에이전트가 했다"는 말이 신뢰를 얻는다.

루프의 핵심은 두 곳의 되돌아가기 화살표다: 피드백에서 Critical/Important를 발견하면 receiving-code-review로, 검증 출력에서 실패를 발견하면 구현으로 돌아간다. "다 됐다"는 주장은 오직 증거(명령 출력)를 확인한 뒤에만 허용된다.

핵심 메커니즘 해설

1. Requesting Code Review — 이른 리뷰, 잦은 리뷰

코드 리뷰는 완료 직전에 한 번 하는 것이 아니다. requesting-code-review의 핵심 원칙은 "Review early, review often"이다 (skills/requesting-code-review/SKILL.md:10). 의무 발동 조건은 세 가지다: 서브에이전트 주도 개발에서 각 태스크 완료 후, 주요 기능 완료 후, main 머지 전(skills/requesting-code-review/SKILL.md:14-17).

리뷰어 서브에이전트는 오케스트레이터의 세션 히스토리를 받지 않는다. 정확히 구성된 컨텍스트만 전달된다: 무엇을 만들었는지 설명(DESCRIPTION), 요구사항(PLAN_OR_REQUIREMENTS), 변경 범위를 나타내는 두 개의 git SHA (BASE_SHA, HEAD_SHA). 이 구조가 리뷰어를 사고 과정이 아닌 작업 결과물에 집중하게 만든다.

피드백 등급별 행동: Critical은 즉시 수정, Important는 다음 단계 진행 전 수정, Minor는 나중을 위해 기록. 리뷰어가 틀렸을 때는 기술적 근거로 반박한다.

2. Receiving Code Review — 기술적 평가, 감정적 수행 아님

피드백을 받는 쪽의 가장 흔한 실수는 맹목적 수행과 과도한 동의 표현이다. "You're absolutely right!", "Great point!" 같은 반응은 명시적으로 금지된다 (skills/receiving-code-review/SKILL.md:29-32). 코드 자체가 피드백을 들었음을 보여주어야 한다 — 말이 아니라 수정으로.

불명확한 항목이 하나라도 있으면 구현을 시작하지 않는다. 항목들이 서로 연관될 수 있기 때문에 부분적 이해 = 잘못된 구현이다 (skills/receiving-code-review/SKILL.md:43-47). 외부 리뷰어의 제안은 "이 코드베이스에 기술적으로 맞는가"를 먼저 검증하고, 기존 기능을 깨는지, 미사용 기능인지(YAGNI)를 확인한 뒤에 반영한다.

6단계 응답 패턴은 감정을 배제하고 기술적 흐름만 남긴다: READ(전체 수신) → UNDERSTAND(재진술 또는 질문) → VERIFY(코드베이스 확인) → EVALUATE(기술적 타당성) → RESPOND(인정 또는 반박) → IMPLEMENT(항목별, 테스트별).

3. Verification Before Completion — 증거 없는 주장 = 거짓말

이 스킬의 출발점은 단호하다: "Claiming work is complete without verification is dishonesty, not efficiency." (skills/verification-before-completion/SKILL.md:10). 핵심 원칙은 "Evidence before claims, always." 규칙의 문자뿐 아니라 정신도 지켜야 한다는 단서까지 달려 있다.

철칙(Iron Law)은 하나다: "NO COMPLETION CLAIMS WITHOUT FRESH VERIFICATION EVIDENCE." 이번 메시지에서 검증 명령을 실행하지 않았다면, 통과한다고 주장할 수 없다 (skills/verification-before-completion/SKILL.md:18-22).

게이트 함수(Gate Function)는 5단계다: IDENTIFY(어떤 명령이 이 주장을 증명하는가) → RUN(전체 명령 실행, fresh) → READ(출력 전체, exit code, 실패 수 확인) → VERIFY(출력이 주장을 뒷받침하는가) → ONLY THEN: 주장. 어느 단계를 건너뛰어도 검증이 아니라 거짓말이다.

주목해야 할 합리화 패턴: "지금쯤이면 작동할 것", "자신 있다", "린터가 통과했으니 빌드도 됐겠지", "에이전트가 성공이라 했다" — 이것들은 모두 검증을 대체하지 못한다. 피로("I'm tired")도 예외가 아니다 (skills/verification-before-completion/SKILL.md:63-76).

4. Finishing a Development Branch — 검증 후 정리

브랜치 마무리 스킬의 핵심 원칙은 "Verify tests → Detect environment → Present options → Execute choice → Clean up." (skills/finishing-a-development-branch/SKILL.md:12). 순서가 엄격하다: 테스트가 통과해야만 다음 단계인 환경 감지와 옵션 제시로 넘어갈 수 있다.

Step 2에서 환경을 감지하는 이유는 normal repo와 worktree에서 사용 가능한 옵션과 정리 방식이 다르기 때문이다. 일반 저장소와 named-branch worktree에서는 4가지 옵션(로컬 머지, PR 생성, 유지, 버리기), detached HEAD에서는 3가지 옵션(PR 생성, 유지, 버리기)을 정확히 제시한다 (skills/finishing-a-development-branch/SKILL.md:68-93).

정리(cleanup)는 Option 1(로컬 머지)과 Option 4(버리기)에만 적용된다. Option 2(PR 생성)와 Option 3(유지)은 worktree를 보존한다 — PR 피드백을 반영하려면 worktree가 살아있어야 하기 때문이다. 버리기(Option 4)는 "discard"를 직접 타이핑하는 확인을 요구한다.

실제 발췌

"직접 만든다면" — 검증 증거를 강제하는 체크리스트

이 네 스킬에서 가장 실용적인 교훈은 하나다: 에이전트의 "완료"를 믿으려면 체크리스트가 아니라 증거를 요구해야 한다. 체크리스트는 에이전트가 스스로 체크할 수 있다 — 실행 없이도. 증거(명령 출력, 테스트 결과, diff)는 위조가 훨씬 어렵다.

환각성 완료 주장을 줄이는 체크리스트 설계

직접 에이전트 워크플로를 설계한다면, 각 완료 주장 앞에 다음을 요구하는 게이트를 넣어야 한다:

  1. 테스트 출력 첨부: "테스트가 통과합니다"가 아니라 테스트 명령 출력 전체(총 몇 개, 실패 0개, exit 0)를 응답에 포함시킨다. 출력 없는 통과 주장은 수락하지 않는다.
  2. 린터와 빌드를 별도로: 린터 통과는 빌드 성공을 의미하지 않는다. 빌드 성공은 테스트 통과를 의미하지 않는다. 각각을 별도 명령으로 실행하고 출력을 확인한다.
  3. 에이전트 성공 보고 ≠ 검증: 서브에이전트가 "성공"이라고 보고해도 VCS diff를 확인하기 전까지는 실제 변경 여부를 알 수 없다. diff가 증거다.
  4. 피드백 항목 수 명시: 코드 리뷰 피드백을 받으면 Critical N개, Important N개, Minor N개를 명시하고 각각의 처리 계획을 밝힌다. "알겠습니다, 수정하겠습니다"는 처리가 아니다.

pushback 문화를 설계에 포함하라

에이전트가 모든 피드백을 수락한다면 그것도 문제다. receiving-code-review는 "외부 피드백 = 따라야 할 명령이 아니라 평가할 제안"으로 규정한다 (skills/receiving-code-review/SKILL.md:209). 에이전트가 pushback을 명시적으로 할 수 있도록 워크플로에 "기술적 반박 허용"을 설계해야 한다. 감정적 수용도, 맹목적 거부도 아닌 — 기술적 근거 있는 반박.

반박할 때는 감정이 아니라 코드/테스트/참조를 제시한다. 반박했다가 틀렸으면 길게 사과하지 않고 "확인했습니다, 수정합니다"로 간결하게 처리한다.

브랜치 마무리 절차를 워크플로의 마지막 게이트로

많은 팀이 구현 완료 후 머지를 급하게 진행한다. finishing-a-development-branch의 교훈은 테스트 → 환경 감지 → 명시적 선택 → 실행 → 정리의 순서를 건너뛰지 말라는 것이다. 특히 worktree를 삭제하기 전에 머지가 성공했는지 검증하고, 버리기 전에 명시적 확인("discard" 타이핑)을 요구하는 패턴은 실수로 인한 작업 손실을 막는 최소한의 안전망이다.

PR 생성 후 worktree를 보존하는 것도 중요하다. 피드백을 받고 반영하려면 worktree가 살아있어야 하기 때문에, PR이 머지될 때까지 worktree를 삭제하지 않는다.

핵심 교훈: 에이전트의 "완료"를 신뢰할 수 있게 만드는 것은 에이전트의 성실함이 아니라 증거를 요구하는 구조다. 명령 출력, diff, 테스트 결과를 완료 주장의 필수 조건으로 만들면 환각성 완료 주장은 구조적으로 줄어든다. 신뢰는 주장이 아니라 증거 위에 세워진다.
