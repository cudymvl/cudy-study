Module 0

전체 아키텍처 조감도

openclaude는 어떤 레이어들로 구성되어 있고, 각 레이어는 무슨 책임을 지는가

왜 이 개념인가

에이전트 코딩 CLI를 처음 보면 "그냥 LLM API를 호출하는 래퍼 아닌가?" 싶을 수 있다. 실제로는 그렇지 않다. openclaude는 여섯 개의 명확한 레이어로 나뉘어 있으며, 각 레이어는 독립적으로 테스트·교체·확장될 수 있도록 설계되어 있다.

이 분리가 왜 필요한가? 에이전트 CLI는 동시에 여러 부담을 안는다. 터미널 UI 렌더링, API 스트리밍, 파일시스템 조작, 권한 게이팅, MCP 프로토콜 통신, 세션 상태 유지. 이 모두를 하나의 파일에 넣으면 어느 하나를 바꿀 때 전체가 흔들린다. 모놀리식 접근 대신 openclaude는 다음 원칙으로 분리한다:

  • 진입점(entrypoints)은 런타임 환경(CLI, SDK, MCP 서버)만 안다 — 대화 로직 없음.
  • 부트스트랩(bootstrap)은 프로세스 생애주기 전체에 걸쳐 살아있는 전역 세션 상태만 담는다 — UI나 API 지식 없음.
  • 엔진(QueryEngine)은 대화 루프를 소유하지만, "어떤 툴이 있는가"는 주입받는다 — 툴 구현 모름.
  • 툴 레지스트리(tools)는 실행 가능한 기능을 모두 알지만, 루프를 모른다 — 엔진 모름.
  • 서비스(services)는 API, MCP, 압축, 권한 같은 인프라만 책임진다 — UI 모름.
  • UI(components)는 React/Ink로 터미널을 그린다 — 비즈니스 로직 없음.
핵심 통찰: 레이어 분리는 코드 구조 문제가 아니라 무엇이 무엇을 모르게 할 것인가의 문제다. 각 레이어 사이의 "모름"이 바로 독립적인 확장과 테스트를 가능하게 하는 격벽이다.

핵심 메커니즘 해설

레이어별 책임

진입점 레이어 — src/entrypoints/

cli.tsx의 main()은 Node.js 프로세스가 시작되자마자 실행된다. 이 레이어의 유일한 책임은 어떤 경로로 시작되었는가를 파악하고 올바른 다음 단계로 라우팅하는 것이다. --version, ps/logs/attach, --bg, daemon, 일반 REPL 등 10개 이상의 fast-path 분기가 있으며, 각 분기는 동적 임포트로 필요한 모듈만 로드한다. "아무것도 모른 척"이 핵심 — 이 레이어는 대화 로직을 직접 실행하지 않는다.

부트스트랩 레이어 — src/bootstrap/state.ts

프로세스 전체 생애를 통틀어 하나만 존재하는 STATE 싱글턴을 제공한다. 세션 ID, 현재 작업 디렉터리, 누적 비용·토큰, 권한 모드, 모델 오버라이드 등 어디서든 읽어야 하는 값들이 여기에 산다. import DAG의 가장 아래 잎(leaf)이므로 다른 어떤 레이어도 임포트하지 않는다. 이 제약 덕분에 순환 의존성 없이 모든 레이어가 상태를 읽을 수 있다.

main.tsx — 초기화 오케스트레이터

실제 애플리케이션 초기화를 수행한다: 설정 로딩, MCP 서버 연결, 툴 목록 조립(getTools()), 권한 컨텍스트 구성, Ink UI 마운트, 대화 이력 복구. QueryEngine에 주입할 QueryEngineConfig를 최종 조립하는 것이 핵심 역할이다.

QueryEngine — 에이전트 루프 — src/QueryEngine.ts

"한 대화 스레드"를 소유하는 클래스다. submitMessage()를 호출하면 시스템 프롬프트 조립 → API 스트리밍 → 툴 실행 → 결과 피드백 → 반복을 담당하는 AsyncGenerator를 반환한다. 중요한 점은 QueryEngine이 툴이 무엇인지 모른다는 것이다 — QueryEngineConfig로 주입받은 Tools 배열과 canUseTool 함수를 호출할 뿐이다. 이 덕분에 헤드리스 SDK 경로와 REPL 경로가 같은 엔진을 공유할 수 있다.

툴 레지스트리 — src/tools/ + src/Tool.ts

Tool<Input, Output> 인터페이스를 만족하는 객체들의 집합이다. 각 툴은 inputSchema(Zod), call(), isEnabled(), isConcurrencySafe(), isReadOnly()를 구현한다. tools.ts의 getAllBaseTools()가 활성화된 모든 툴을 배열로 반환하며, 이것이 QueryEngine에 주입된다. 툴은 엔진을 알지 못한다.

서비스 레이어 — src/services/

API 호출(api/), MCP 통신(mcp/), 컨텍스트 압축(compact/), 권한 정책(policyLimits/), 분석(analytics/) 등 인프라 기능을 캡슐화한다. 이 레이어는 UI나 루프 구현을 모른다 — 순수하게 "주어진 요청을 처리"한다.

UI 레이어 — src/components/

React + Ink(터미널 UI 프레임워크)로 동작한다. 스트리밍 출력 렌더링, 권한 대화상자, 진행 표시기, 메시지 히스토리 표시를 담당한다. UI는 submitMessage()를 통해 QueryEngine에 입력을 보내고, QueryEngine이 emit하는 SDKMessage 스트림을 소비해 화면을 갱신한다.

제어 흐름 vs 데이터 흐름

제어 흐름(누가 누구를 호출): CLI → bootstrap → main → QueryEngine → Tool / Provider / Permission 순서로 호출이 아래로 내려간다. UI는 사용자 입력을 받아 다시 QueryEngine으로 올려 보내는 반환 경로다.
데이터 흐름(메시지·토큰이 어디로): 사용자 텍스트 → QueryEngine이 메시지 배열에 추가 → API로 전송 → 스트리밍 토큰이 QueryEngine에서 SDKMessage로 변환 → UI가 렌더 → 툴 결과가 다시 메시지 배열로 피드백. STATE 싱글턴은 이 흐름 바깥에서 누적 비용·토큰 등 집계 메타데이터만 기록한다.

실제 코드 예시

"직접 만든다면" 교훈

최소 에이전트 CLI 레이어 의사코드

openclaude의 레이어 분리를 가장 단순하게 재현하면 다음과 같다:

// Layer 1: Entrypoint — only knows "which path am I on?"
async function cliMain(args: string[]) {
  if (args[0] === '--version') { print(VERSION); return }
  await appMain()  // dynamic import
}

// Layer 2: Bootstrap — global singleton, no imports from above
const STATE = { sessionId: uuid(), cwd: process.cwd(), cost: 0 }

// Layer 3: Orchestrator — assembles dependencies, mounts UI
async function appMain() {
  const tools = getTools()          // tool registry
  const canUseTool = makePermissionGate()
  const engine = new QueryEngine({ tools, canUseTool, ... })
  mountUI(engine)  // pass engine to UI, not the other way
}

// Layer 4: Engine — owns conversation loop, knows nothing about tools
class QueryEngine {
  async *submitMessage(prompt: string) {
    const messages = [...this.history, userMsg(prompt)]
    for await (const chunk of callAPI(messages, this.tools)) {
      if (chunk.type === 'tool_use') {
        const tool = this.tools.find(t => t.name === chunk.name)
        const result = await tool.call(chunk.input, this.context)
        messages.push(toolResult(result))
        // loop continues — engine sends tool result back to API
      }
      yield chunk
    }
  }
}

// Layer 5: Tool — knows its job, knows nothing about the loop
const BashTool: Tool = {
  name: 'Bash',
  call: async ({ command }, ctx) => {
    const ok = await ctx.canUseTool(BashTool, { command })
    if (!ok) return { data: 'Permission denied' }
    return { data: await exec(command) }
  }
}

엔진과 툴을 분리하는 이유 — 트레이드오프 분석

분리 방식 장점 비용
엔진이 툴을 직접 알 때
(모놀리식)		 코드가 한 곳에 있어 단기적으로 단순 툴 추가마다 엔진 수정 필요, 테스트 격리 불가
주입(Injection) 방식
(openclaude 채택)		 SDK/REPL/테스트가 같은 엔진 재사용.
툴 Mock 주입으로 엔진 단독 테스트 가능.
feature flag로 툴 활성화/비활성화 가능. 		설정 조립 코드(main.tsx) 복잡도 증가
툴이 엔진 호출 시
(역전)		 툴이 능동적으로 대화 제어 가능 순환 의존성, 디버깅 지옥, 권한 게이트 우회 위험
직접 만들 때 가장 많이 틀리는 부분: "툴 결과를 어떻게 엔진에 돌려보내지?" — openclaude의 답은 툴이 결과를 반환하면 엔진이 그것을 메시지 배열에 추가한 뒤 다시 API를 호출하는 것이다. 이것이 "에이전트 루프"다. 툴은 API를 절대 직접 호출하지 않는다.

bootstrap을 import DAG의 leaf로 두는 이유

state.ts가 다른 src 파일을 임포트하지 않는 규칙은 단순해 보이지만 의미가 크다. 어떤 레이어든 STATE를 읽어야 할 때 순환 의존성 없이 bootstrap/state를 임포트할 수 있다. 만약 state.ts가 서비스나 UI를 임포트한다면 초기화 순서 문제(circular init)가 발생해 Node.js 모듈 시스템이 undefined를 반환한다. 실제 코드에는 // bootstrap-isolation forbids importing...라는 ESLint 규칙도 있다.
